---

## 구조적 방법론과 Agile 방법론 비교

| 구분 | 구조적 방법론 | Agile 방법론 |
|:-----|:-------------|:-------------|
| **개념** | - 절차 중심의 구조중심 분석/설계 그리고 문서화 모듈화 중심의 소프트웨어 개발 방법론 | - 상호협력 및 내/외부 변화에 대한 민첩한 대응을 추구하는, 반복점진적 소프트웨어 개발 방법론 |
| **개념도** | `요구사항 분석 → 구조적 분석 → 구조적 설계 → 구조적 프로그래밍` | `1단계 ⟷ 2단계 ⟷ 3단계` (반복적 순환) |
| **특징** | - 요구사항 분석및 요구사항 추출 기법 활용<br>- 기능 중심, 모듈의 응집도와 결합도 고려<br>- Top-Down 방식의 업무활동 구체화 | - 초기테스트, 의사소통, 상호작용 중시<br>- 고객과 개발팀의 소통중심 분석&설계<br>- Bottom-Up 방식의 SW구체화 |
| **관점** | - 기능 중심<br>- 분할과 정복 | - 고객과의 지속적인 협력<br>- 작동하는 소프트웨어 구현 |
| **기법** | - 정형화<br>- 모듈화<br>- 사용자 요구사항 문서화 | - XP<br>- SCRUM<br>- KANBAN<br>- LEAN |
| **도구** | - Data Flow Diagram<br>- Data Dictionary<br>- State Transition Diagram | - CI/CD<br>- 테스트 자동화, 커버리지 도구<br>- TDD를 위한 테스트 구현 도구 |

## Agile 선언문 (Agile Manifesto)

### 4가지 가치

| 기존 가치 | | Agile 가치 |
|:---------|:-:|:----------|
| 프로세스와 도구 | < | **개인과 상호작용** |
| 포괄적인 문서화 | < | **작동하는 소프트웨어** |
| 계약 협상 | < | **고객과의 협력** |
| 계획을 따르는 것 | < | **변화에 대응하는 것** |

---

## 12가지 원칙

1. 고객 만족을 위한 가치 있는 소프트웨어의 조기/지속적 인도
2. 요구사항 변경 환영 (후기 개발 단계에서도)
3. 작동하는 소프트웨어를 자주 인도 (2주~2개월)
4. 비즈니스 담당자와 개발자의 매일 협업
5. 동기 부여된 개인들로 프로젝트 구성
6. 대면 대화가 가장 효율적인 의사소통
7. 작동하는 소프트웨어가 진척의 주된 척도
8. 지속 가능한 개발 (일정한 속도 유지)
9. 기술적 우수성과 좋은 설계에 대한 지속적 관심
10. 단순성 (하지 않아도 되는 일 최소화)
11. 자기 조직화 팀
12. 정기적으로 효율성 개선 방법 반영

## Agile 방법론 종류

| 방법론 | 특징 |
|:-------|:-----|
| **XP (eXtreme Programming)** | 짝 프로그래밍, TDD, 지속적 통합, 리팩토링 |
| **Scrum** | 스프린트, 데일리 스크럼, 스프린트 리뷰, 회고 |
| **Kanban** | 시각화, WIP 제한, 흐름 관리 |
| **Lean** | 낭비 제거, 가치 흐름, 지속적 개선 |
