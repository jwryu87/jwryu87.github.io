# 명세 기반 테스트 (Specification Based Test)
{: .fs-8 }

9.2 테스트 기법
{: .label .label-yellow }

---

## 핵심 키워드

`동등값 분할` `경계값 분석` `의사결정 테이블` `상태전이 테스팅` `유스케이스` `페어와이즈`

---

## 정의/개념

테스트 케이스를 프로그램의 요구사항 명세서를 기반으로 테스트 케이스를 선정하여 테스트하는 기법

---

## 블랙박스 테스트의 기법 설명

| 기법 | 개념도 | 상세 설명 |
|:-----|:-------|:---------|
| **동등 클래스 분할 기법<br>(Equivalence Class Partitioning)** | 입력범위: 0 ~ 40 ~ 70 ~ 100 | - 프로그램의 입력 도메인을 동가영역들로 분할 후 각 영역별로 대표되는 값들을 선정하여 테스트케이스를 설계하는 방법<br>- 예) 입력데이터 x값이 0 ~ 100이면 경우의 TC를 (x < 0), (x > 50), (x > 100)으로 분할하여 적용 |
| **경계값 분석<br>(Boundary Value Analysis)** | 경계 / 경계 / 경계 | - 입력 영역의 분할 클래스의 경계값으로 테스트 케이스를 설계하는 방법<br>- 동가 분할 기법의 확장, 동가분할 된 경계의 유효한 값, 경계의 가장 가까운 유효하지 않은 값 선택 대상 테스트 진행<br>- 예) x값이 0~100사이여야 한다면 TC를 (x=0), (x=100), (x<0:01), (x=100.1)으로 정의 |
| **페어와이즈 테스팅<br>(Pairwise Testing)** | Test Cases 매트릭스 | - 대부분 결함이 2 개의 요소(Pair)의 상호작용에서 기인한다는 점에 착안하여, 2개 값들이 다른 파라미터의 값과 최소 한번씩은 조합을 이루도록 구성된 테스트 기법<br>- 모든 조합을 포함하지 않으므로 결함을 찾지 못하는 경우 발생 가능<br>- 입력 데이터간 관계가 출력에 영향을 미치는 상황을 체계적으로 분석하여 테스트 케이스 설계 및 테스트 |
| **원인-결과 그래프<br>(Cause Effect Graph)** | 입력조건 → 결과/액션 | - 원인,결과에 근거한 테스트 케이스 작성하여 시스템 외부 동적인 고려<br>- 관련 시스템 내부 입력 상태 결과, 시스템 분할 또는 동일 조합으로 인한 출력 결과 |
| **의사결정 테이블 테스팅<br>(Decision Table Testing)** | 조건 조합 표 | - 주요한 의사결정 요소들을 표(결정테이블)로 만들고, 요소들간의 결합에 의한 테스트 케이스 설계<br>- 각 의사결정 요소들의 조합을 통해 다양한 형태의 테스트 시나리오를 도출 |
| **상태 전이 테스팅<br>(State transition Testing)** | 상태 다이어그램 | - 상태전이 다이어그램 통해 이벤트, 액션, 활동, 상태 변화로 발생되는 관계, 동작 파악하여 테스트<br>- 임베디드SW 테스트시 적용 |
| **유스케이스 테스팅<br>(Use Case Testing)** | Use Case 다이어그램 | - 유즈케이스를 통해 도출되는 비즈니스 시나리오/기본 흐름(normal flow) 기반으로 테스트<br>- 팁/선/팁/다른 유형 유즈케이스 테스트<br>- 시스템 레벨 유즈케이스 테스트 |
| **분류 트리 기법<br>(Classification Tree Method)** | 트리 구조 | - SW의 일부 또는 전체를 트리 구조로 분석 및 표현하여 테스트 케이스를 설계하는 기법<br>- 트리구조를 시각화 하여 테스트 케이스를 설계하므로 불필요한 장복 및 누락을 최대 |
| **오류예측 기법<br>(Error Guessing)** | 직관/경험 기반 | - 각 시험 기법들이 놓치기 쉬운 오류들을 감각과 경험으로 찾아 검증<br>- Ad-hoc Testing이라고도 하며, 직관과 경험에 의한 특정 형태의 결함 예측 및 해당 결함을 드러내는 테스트 케이스 설계 기법 |

- 화이트박스 테스트와 블랙박스 테스트의 중점 부분 테스트를 위한 그레이박스 테스트도 활용 가능
