# 리틀의 법칙 (Little's Law)
{: .fs-8 }

9.3 테스트 종류
{: .label .label-green }

---

## 핵심 키워드

`용자가 증가함에 따라 응답시간이 느려지고 TPS는 증가`

---

## 정의/개념

공간 내 머무는 객체수는 객체의 공간 유입량과 객체 머무는 시간에 비례 법칙

---

## 리틀의 법칙 공식

$$
L = \lambda \times W
$$

| 변수 | 설명 | 성능테스트 적용 |
|:-----|:-----|:---------------|
| **L** | Average number of items in a system<br>(공간 내에 머무는 객체수) | 응답수 또는 처리수 |
| **λ** | Average arrival rate<br>(객체의 공간 유입량) | TPS (초당처리시간) |
| **W** | Average wait time in the system for an item<br>(객체가 머무는 시간) | MRT (평균응답시간) |

{: .note }
> 리틀의 법칙은 MIT가 개설한 경영과학 과정에서 최초로 학위를 받은 존 리틀에 의해 만들어진 경제관련 법칙

---

## 성능테스트 적용

- (정의) 공간 내에 머무는 객체수(λ)는 객체의 공간 유입량(W)과 객체가 머무는 시간(R)에 비례한다는 법칙
- 사용자가 증가함에 따라 응답시간이 느려지고 TPS(초당처리시간)는 증가
- TPS(초당처리시간) = CU(동시사용자) / Request Interval(응답시간 + Think Time)
- TPS가 더이상 증가하지 않고 완만하게 되는 시점이 그 시스템의 임계치

```
응답수
또는    │
처리수  │                    _______________
        │                 ___/                (임계치)
        │              __/
        │           __/
        │        __/
        │     __/  TPS
        │   _/     (초당처리시간)
        │ _/
        │/
        └────────────────────────────────────
                                      테스트 시간
```

---

## 리틀의 법칙을 통한 성능테스트의 목적

| 목적 | 설명 | 예시 |
|:-----|:-----|:-----|
| **수학적 근거 테스트** | - TPS = AU/MRT 라는 수학적 모델에 근거하여 테스트 수행 | 대상 시스템의 정량지표 마련 |
| **시스템의 성능 및 용량관리** | - 성능의 임계치 및 허용 가능한 수준의 용량 파악하여 시스템운영의 가용성 및 효율성 증가 | 시스템 성능 품질 확보 |
| **장애 예방** | - 과부하가 예상되는 시스템의 임계치를 추적, 사전 장애 예방 | 시스템의 사용성 확보 |
| **리드타임 성과측정** | - LeadTime = WIP(Work In Process) / ACR(Average Completion Rate)<br>- 하나의 프로세스가 시스템을 통과하는 시간의 측정을 통해 부가가치 시간을 증가 가능 | 생산성을 향상시키고, 기업의 KPI 수립 및 CSF 달성 |

---

## 성능테스트와 리틀의 법칙 관계

{: .highlight }
> 성능테스트는 리틀의 법칙을 통해 **업무량, 응답시간, 시스템 사용률**을 측정하고 프로젝트 성능 여부의 도달여부를 검증하는데 가장 큰 목적

- 리틀의 법칙은 경제관련 법칙에서 출발하였으나, 정보시스템의 성능테스트를 위한 임계치 보장, 부하도, 스트레스 측정 등을 위해 시스템의 최대 발휘 성능을 측정하고 장애를 대응하는데 적용 가능
