## IEEE829 기반 테스트케이스 구성요소

| 구성요소 | 설명 | 역할 |
|:---------|:-----|:-----|
| **식별자** | - Test case specification identifier<br>- 테스트케이스를 식별하기 위한 유일한 식별자 | 추적성 |
| **테스트 항목** | - Test items<br>- 테스트할 모듈이나 기능 | 테스트 대상 |
| **입력 명세** | - Input specifications<br>- 입력 값 또는 조건 | 입력 값 |
| **출력 명세** | - Output specifications<br>- 테스트케이스 실행 시 기대되는 결과 | 출력 값 |
| **환경 설정** | - Environmental needs<br>- 테스트 수행할 때 필요한 HW/SW 환경 | 테스트 베드 |
| **특수절차요구** | - Special procedural requirements<br>- 테스트케이스 수행 시 특별히 요구되는 절차 | 프로세스 |
| **의존성 기술** | - Intercase dependencies<br>- 테스트케이스 간의 의존성 | 선/후 관계 |

---

## IEEE-829 기반 테스트 케이스 설계절차

| 순서 | 절차 | 설명 |
|:-----|:-----|:-----|
| 1 | 테스트 케이스 식별 | 프로젝트 특성, 테스트 전략, 테스트 계획서 기반으로 식별 작성 |
| 2 | 테스트 스텝 및 검증 포인트 정의 | 테스트 케이스 기반, 시나리오와 성공/실패의 기준 정의 |
| 3 | 테스트 케이스 속성 정의 | 테스트 케이스 별 작성자, 작성일, ID, 설명 등의 속성을 정의 |
| 4 | 테스트 데이터 준비 및 검증 | 상위레벨 테스트 케이스들을 실행 가능한 형태의 데이터 준비 |
| 5 | 테스트 케이스 명세서 작성 | 연관성 있는 테스트 케이스를 순서에 맞게 구성 |
| 6 | 테스트 케이스 명세서 검토 | 프로젝트 관리자 및 품질 보증 담당자 동과 검토 및 협의 수행 |
| 7 | 테스트 케이스 명세서 승인 | 작성된 테스트 케이스에 대한 승인 요청 |

{: .note }
> 테스트 케이스 설계는 테스트 이전에 이루어져야 하고, 이상적으로는 시스템 설계 시 작성 필요
