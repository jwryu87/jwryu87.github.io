## 정의/개념

실 서비스에 장애를 주입하여, 출시 전 테스트에서 드러나지 않은 **아키텍처상의 문제를 테스트하기 위한 테스트 기법**

---

## 프로세스

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ 정상상태 │──▶│ 가설수립 │──▶│실험디자인│──▶│ 결과확인 │──▶│문제점수정│
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
 정상지표측정   시나리오설계    테스트계획    실행및가설검증   수정및개선
                                                
                    ┌─────────────────────────────┐
                    │   Chaos Engineering Team     │
                    │     테스트 도구 및 자동화     │
                    └─────────────────────────────┘
```

---

## 프로세스 상세

| 단계 | 활동 | 설명 |
|:-----|:-----|:-----|
| **1. 정상상태** | 정상 지표 측정 | 시스템의 측정 가능한 값을 이용해 정상적인 동작 상태를 정략적으로 측정 (CPU로드, NW I/O) |
| **2. 가설수립** | 시나리오 설계 | 항상 정상적인 상태를 유지한다는 가설 기반으로 주요 테스트 시나리오 작성 (DB 다운, 접속초과 DDoS 공격) |
| **3. 실험디자인** | 테스트 계획 | 실험 가설 선택, 실험범위/규칙 설정 측정지표 선정, 실험계획 알림 (롤백계획, 작업범위) |
| **4. 결과 확인** | 실행 및 가설 검증 | 정상 지표와 테스트 결과를 비교하여 가설 검증 및 문제점 확인 (장애 감지 시간, 전파/알림 시간) |
| **5. 문제점 수정** | 수정 및 개선 | 발생된 문제점에 대해 문제점을 수정 및 지속적으로 개선 (분간 접속 지연 발생, SLA 미충족) |

---

## 수행방안

### 1) 팀구축
- 카오스 엔지니어링 팀 구성

### 2) 도구 활용
- **Chaos Monkey**: Netflix 개발, EC2 인스턴스 무작위 종료
- **Cube Monkey**: Kubernetes 환경용
- **GameDay**: AWS 제공 카오스 엔지니어링 이벤트
- **Failure Injection**: 장애 주입 도구
- **ChAP**: Chaos Automation Platform
- **Gremlin**: 상용 카오스 엔지니어링 플랫폼

### 3) Layer 별 테스트
| Layer | 설명 |
|:------|:-----|
| 개발팀 | 애플리케이션 레벨 테스트 |
| 미들웨어 | 미들웨어 장애 테스트 |
| 서버 | 서버 장애 테스트 |
| 네트워크 | 네트워크 장애 테스트 |
| 회선 | 회선 장애 테스트 |

{: .highlight }
> 카오스 엔지니어링 팀 구성 및 자동화 도구 통한 테스트 및 지속적 훈련 수행

---

## 카오스 엔지니어링의 원칙

1. **정상 상태 행동에 대한 가설 수립**
2. **실제 세계의 이벤트 변동**
3. **프로덕션에서 실험 수행**
4. **지속적 실험 자동화**
5. **폭발 반경 최소화**
