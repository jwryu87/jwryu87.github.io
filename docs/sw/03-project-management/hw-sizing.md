## 정의/개념

기본적인 용량과 성능 요구사항이 제시되었을 때, 그것을 **시스템 요구사항으로 변환하는 작업/결과**

---

## 프로세스

| 절차 | 활동 | 설명 |
|:-----|:-----|:-----|
| **[1단계]<br>구축방향 및 기초자료 조사** | 구축 방향 파악 | - 전체 시스템에 포함되는 대략적 서버 수, 통신 환경 파악<br>- APP 아키텍처(2계층,3계층), 업무 성격, 정보 흐름 파악 |
| | 기초자료 조사 | - 공통/OLTP: 응답속도, 최대 사용자, 업무 내용 등 조사<br>- WEB/WAS: 시스템 용도, 서비스 형태, 사용자 수 등 조사 |
| **[2단계]<br>기준 부하 산정** | 기초자료 및 업무분석 | - OLTP: 동시 사용자 수 및 분당 트랜잭션 수(TPM) 산정<br>- WEB/WAS: 동시 사용자 및 세션 수, 단위 오퍼레이션 산정<br>- 비즈니스 요구사항 포함, 트랜잭션 타입, 특성, 가중치 조사<br>- 온라인/배치 구분, 요구 시간, 처리 볼륨, 복잡성 분석 |
| **[3단계]<br>참조모델 결정 및 서버 규모산정** | 참조 모델 선택 | - 참조모델1(WEB/WAS/OLTP): 단일 서버에서 모든 계층 처리<br>- 참조모델2(WEB/WAS, OLTP): 응용 계층과 DB 계층 분리<br>- 참조모델3(WEB, WAS, OLTP): 각 응용 계층과 DB 계층 분리 |
| | 요소별 규모산정 | - 2단계에서 조사된 업무 분석 자료 보정 계수 산정<br>- CPU, 메모리, 디스크 구성요소에 대해 각각 규모 산정 수행 |
| **[4단계]<br>참조모델 별 가중치 적용** | 참조모델1 가중치 적용 | - WEB/WAS/DB 역할을 동시에 수행하므로 서버의 CPU 규모에<br>- 상대적 서버 가중치 2.1(WEB:0.4, WAS:0.7, DB:1) 적용 |
| | 참조모델2 가중치 적용 | - WEB/응용서버, DB서버: WEB/응용서버의 경우 WEB, WAS의 역할을 동시에 수행해야 하므로 1.6(WEB:0.6, WAS:1) 적용<br>- WEB서버, 응용/DB서버: 응용/DB서버의 경우 응용과 DB서버의 역할을 동시에 수행해야 하므로 1.7(응용:0.7, DB:1) 적용 |
| | 참조모델3 가중치 적용 | - 별도의 서버 가중치 미적용<br>- WEB: 1, WAS: 1, DB: 1 |

{: .note }
> 일반적으로 규모 산정의 정확도는 규모 산정을 위한 기초 자료 확보, 표준화된 계산식과 절차, 산정 결과의 데이터베이스 저장 및 재활용 등의 4가지 요소에 의해서 결정

---

## 규모산정방법

| No | 규모산정 방법 | 개념 상세설명 | 특징 |
|:---|:------------|:------------|:-----|
| ① | **수식계산법** | - 동접자 50 명 → VM 2core<br>- 동접자 5 만명 → VM 64core<br>- 사용자 수등 규모산정을 위한 요소를 토대로 용량수치를 계산하고, 보정치를 적용하는 방법 | - 기존자료 분석 기반<br>- 수식활용 |
| ② | **참조법** | - 동접자 9,000 천명 규모산정<br>- 참조 REPOSITORY 1 만명 규모산정 데이터<br>- 업무량(사용자 수, DB 크기)에 따라 기본데이터를 토대로 대략적인 시스템 규모를 비교하여 비슷한 규모를 산정하는 방법 | - 유사정보 참조<br>- 재활용<br>- 재사용 |
| ③ | **시뮬레이션법** | - <업무 모델링 환경 구축><br>- WEB → WAS → 시뮬레이션 수행 → 규모산정<br>- 대상업무에 대한 작업부하를 모델링하고 시뮬레이션하여 규모를 산정하는 방법 | - 유형별 테스트<br>- 비용투자 필요 |

{: .highlight }
> 정보시스템 하드웨어 규모산정 방법들의 특수성이 존재하여, 개별 장점과 단점을 사전검토 필요
