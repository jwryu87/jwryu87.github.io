### 🔑 핵심 키워드 3개

| 키워드 | 쉬운 설명 |
|:------|:---------|
| **SOAP** | 편지 봉투처럼 정해진 형식으로 데이터 전송 |
| **REST** | URL로 직접 자원에 접근하는 간단한 방식 |
| **API** | 프로그램끼리 대화하는 방법 |

---

### 📖 등장배경

> **문제**: 웹 서비스끼리 통신하려면...
> - 서로 다른 시스템 간에 데이터 교환 필요
> - 표준화된 통신 방법 필요
>
> **해결 1**: SOAP (2000년대 초)
> - XML 기반의 엄격한 프로토콜
> - 대기업, 금융권에서 선호
>
> **해결 2**: REST (2000년대 중반~)
> - HTTP 기반의 간단한 아키텍처
> - 모바일, 웹 서비스에서 인기

---

### 📝 개념 정의

> 🍕 **피자 주문**에 비유하면...
>
> | 방식 | 비유 |
> |:----|:----|
> | **SOAP** | 정해진 주문서 양식에 맞춰서 팩스로 주문 (엄격함) |
> | **REST** | 웹사이트에서 클릭 몇 번으로 주문 (간편함) |
>
> **핵심 차이**: SOAP은 **프로토콜**, REST는 **아키텍처 스타일**

---

### 🏗️ 기술요소 (2그룹)

#### 그룹 1: SOAP의 특징

| 요소 | 설명 |
|:----|:----|
| **XML만 사용** | 데이터 형식이 XML로 고정 |
| **WSDL** | 서비스 명세서 (어떤 기능이 있는지 설명) |
| **UDDI** | 서비스 등록소 (어디에 서비스가 있는지) |

#### 그룹 2: REST의 특징

| 요소 | 설명 |
|:----|:----|
| **다양한 포맷** | JSON, XML, HTML 등 자유롭게 사용 |
| **HTTP 메소드** | GET, POST, PUT, DELETE로 CRUD |
| **Stateless** | 상태를 저장하지 않음 |

---

### ⭐ 차별점 키워드

**Stateless (무상태)** 🔄
> REST의 핵심 원칙으로, 서버가 클라이언트의 상태를 저장하지 않음
> → 서버 확장성 좋고, 캐싱 가능!

---

### 📋 6하원칙 요약

| 구분 | SOAP | REST |
|:----|:----|:----|
| **누가** | 대기업, 금융권 | 스타트업, 모바일 앱 |
| **무엇을** | XML 기반 메시지 전송 | HTTP로 리소스 접근 |
| **언제** | 높은 보안, 트랜잭션 필요 시 | 빠른 개발, 간단한 API 필요 시 |
| **어디서** | 레거시 시스템, ERP 연동 | 웹, 모바일 앱, 마이크로서비스 |
| **왜** | ACID 보장, WS-Security | 가볍고 빠르고 캐싱 가능 |
| **어떻게** | WSDL + UDDI + WS 스택 | HTTP + URL + JSON |

## 🎯 기술사 수준 설명

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **SOAP** | SOA 구조의 XML 기반 프로토콜 |

| **Stateless** | 서버가 클라이언트 상태를 저장하지 않음 |

---

### 📖 등장배경 (2개)

| 배경 | 설명 |
|:----|:----|
| **분산 시스템 통합 필요** | 이기종 시스템 간 표준화된 통신 방법 필요 |
| **웹 서비스 표준화** | W3C 표준 (SOAP) vs 경량화된 접근 (REST) |

---

### 📝 개념 정의 (22자)

|:----|:----|

| **REST** | 중간 매체 없이 리소스를 직접 Stateless로 제공 |

## 📊 SOAP vs REST 상세 비교 `유기데보대캐페`

### 핵심 특성 비교

| 구분 | SOAP | REST |
|:----|:----|:----|
| **유**형 | 프로토콜 (Protocol) | 아키텍처 스타일 (Architectural Style) |
| **기**능 | 기능 위주로 구조화된 정보 전송 | 데이터 위주로 리소스에 접근 |
| **데**이터 포맷 | XML만 사용 가능 | 일반 텍스트, HTML, XML, **JSON** 등 다양 |
| **보**안 | WS-Security와 SSL 지원 | SSL, HTTPS 지원 |
| **대**역폭 | 상대적으로 더 많은 리소스와 대역폭 필요 | 상대적으로 리소스 적게 필요, **가벼움** |
| **캐**시 | 캐시 사용 **불가** | 캐시 사용 **가능** |
| **페**이로드 | 엄격한 통신규약, 모든 메시지는 사전 정의 필요 | 미리 알림 필요 없음 |
| **ACID** | 자체적으로 ACID 기준 있어 데이터 손상 최소화 | ACID 관련 내용 미존재 |

---

## 🔧 적용 기술 비교 `R-S-W-U-W`

| 구분 | SOAP | REST |
|:----|:----|:----|
| **전달 메커니즘** | **R**PC (Remote Procedure Call) | Publish/Syndicate Pattern |
| **전달 프로토콜** | **S**OAP/HTTP, SMTP | HTTP |
| **서비스 명세** | **W**SDL | WADL, XML, JSON, REST |
| **레지스트리** | **U**DDI | 없음 |
| **필요 스택** | **W**3C의 WS-* 스택 | 없음 |

---

### 🧠 암기법 상세

```
🧠 유기데보대캐페 = "SOAP vs REST 7대 비교 항목"

  유 : 유형 (프로토콜 vs 아키텍처 스타일)
  기 : 기능 (기능 위주 vs 데이터 위주)
  데 : 데이터 포맷 (XML only vs JSON 등 다양)
  보 : 보안 (WS-Security vs SSL/HTTPS)
  대 : 대역폭 (많이 필요 vs 가벼움)
  캐 : 캐시 (불가 vs 가능)
  페 : 페이로드 (엄격 vs 자유)

🧠 R-S-W-U-W = "SOAP 기술 스택"

  R : RPC (Remote Procedure Call)
  S : SOAP/HTTP (전달 프로토콜)
  W : WSDL (서비스 명세)
  U : UDDI (레지스트리)
  W : WS-* (필요 스택)
```

## 📊 사용 사례 비교

| 구분 | SOAP | REST |
|:----|:----|:----|
| **적합한 경우** | 높은 보안 필요, 트랜잭션 처리, ACID 보장 | 빠른 개발, 모바일 앱, 마이크로서비스 |
| **산업** | 금융, 의료, 대기업 ERP | 스타트업, 소셜 미디어, 모바일 앱 |
| **대표 사례** | 은행 API, 정부 시스템, 레거시 시스템 | Twitter API, Google API, Facebook API |
| **현재 트렌드** | 레거시 시스템 유지보수 | 신규 프로젝트 대부분 채택 |

## ⭐ 차별점 키워드

### 🏆 RESTful API & Richardson Maturity Model
> REST 원칙을 얼마나 잘 준수하는지를 **4단계 성숙도 모델**로 평가
>
> | Level | 설명 |
> |:------|:-----|
> | **Level 0** | HTTP를 원격 통신 터널로만 사용 (POX) |
> | **Level 1** | 리소스 개념 도입 (개별 URI) |
> | **Level 2** | HTTP 메소드 활용 (GET, POST, PUT, DELETE) |
> | **Level 3** | HATEOAS (하이퍼미디어 컨트롤) |

### 📋 6하원칙 요약

| 구분 | SOAP | REST |
|:----|:----|:----|
| **누가 (Who)** | 대기업, 금융권, 의료 시스템 | 스타트업, 모바일 앱 개발자 |
| **무엇을 (What)** | XML 기반 기능 중심 메시지 전송 | HTTP 기반 리소스 중심 접근 |
| **언제 (When)** | ACID, 보안, 트랜잭션 필요 시 | 빠른 개발, 캐싱, 확장성 필요 시 |
| **어디서 (Where)** | 레거시 시스템, ERP, 은행 | 웹, 모바일, 마이크로서비스 |
| **왜 (Why)** | WS-Security, ACID 보장 | 가볍고 빠르고 유연함 |
| **어떻게 (How)** | WSDL + UDDI + WS-* 스택 | HTTP + URL + JSON |
