### 🔑 핵심 키워드 3개

| 키워드 | 쉬운 설명 |
|:------|:---------|
| **여러 클라우드 동시 사용** | AWS, Azure, GCP 등 여러 곳을 함께 사용 |
| **벤더 종속 방지** | 한 회사에만 의존하지 않음 |
| **장애 대비** | 한 곳이 고장나도 다른 곳으로 대체 |

---

### 📖 등장배경

> **문제**: 한 클라우드만 쓰면...
> - AWS가 장애나면? 🔥 서비스 전체 마비
> - AWS 가격 올리면? 💰 울며 겨자먹기
> - AWS에 없는 기능 필요하면? 🤷 포기
>
> **해결**: 멀티클라우드!
> - 여러 클라우드를 동시에 사용
> - 각 클라우드의 장점만 골라서 조합
> - 한 곳 장애나도 다른 곳으로 자동 전환

---

### 📝 개념 정의

> 🏦 **여러 은행 계좌**와 비슷해요!
>
> | 방식 | 비유 |
> |:----|:----|
> | **단일 클라우드** | 한 은행만 사용 (장애 시 돈 못 찾음) |
> | **멀티클라우드** | 여러 은행 사용 (각 은행 장점 활용) |
>
> **핵심**: 여러 클라우드를 **함께 연결**해서 사용하는 것!

### 🏗️ 기술요소 (2그룹)

#### 그룹 1: 인프라 관리 기술

| 요소 | 역할 |
|:----|:----|
| **인프라 연동** | 여러 클라우드를 하나처럼 연결 |
| **통합 모니터링** | 모든 클라우드를 한 화면에서 관리 |

#### 그룹 2: 애플리케이션 관리 기술

| 요소 | 역할 |
|:----|:----|
| **자동 배포** | 앱을 여러 클라우드에 자동으로 설치 |
| **로드밸런싱** | 트래픽을 여러 클라우드에 분산 |

---

### ⭐ 차별점 키워드

**Cloud Brokerage (클라우드 브로커리지)** 🤝
> 여러 클라우드 중에서 최적의 조합을 찾아주는 **중개 서비스**
> → 마치 보험 비교 사이트처럼!

## 🎯 기술사 수준 설명

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **Lock-in 탈피** | 특정 CSP 벤더 종속성 해소 |
| **Fail-over** | 장애 시 자동 전환 및 고가용성 확보 |
| **Best-of-Breed** | 각 CSP의 최적 서비스 조합 활용 |

---

### 📖 등장배경 (2개)

| 배경 | 설명 |
|:----|:----|
| **단일 클라우드 한계성 극복** | 하나의 클라우드로 모든 기능 구현 불가, 벤더 종속 문제 |
| **글로벌 CSP 확산** | AWS, Azure 등 글로벌 사업자 진출로 국내 사업자 역할 재정립 필요 |

## 🏗️ 구성요소 (2그룹 × 4항목) `인서앱통` `개CLI웹`

### 그룹 1: 멀티클라우드 핵심 기술 `인서앱통`

| 기술 유형 | 주요 기능 | 설명 |
|:---------|:---------|:-----|
| **인**프라 연동 기술 | 단일 API, 연동 드라이버 | 이종 클라우드를 공통 방법으로 연동 및 제어, 자원 라이프사이클 통합 관리 |
| **서**비스 통합 운영 | 적재 배치 스케줄링 | 응용이 요구하는 최적 클라우드 및 자원의 선정, 배치 |
| **앱** 통합 운영 | App 생성/배치/관리 | 다양한 클라우드에 애플리케이션 자동 설치, 이동, 복제, 로드밸런싱 |
| **통**합 모니터링 | Agent, Push/Pull | 대규모 모니터링 데이터 수집/처리, 서비스간 관계 모니터링 |

### 그룹 2: 개방형 인터페이스 기술 `개CLI웹`

| 구성요소 | 설명 |
|:--------|:-----|
| **개**방형 API | REST API, gRPC API 등 업계 표준 응용 프로그래밍용 인터페이스 |
| **CLI** | 터미널 환경의 명령어 인터페이스 |
| **웹**도구 | 웹 환경의 GUI 인터페이스 |

### 📊 멀티클라우드 필요성 `단벤페서`

| 구분 | 필요성 | 설명 |
|:----|:------|:-----|
| **사용자 측면** | **단**일 클라우드의 한계 | 하나의 클라우드로 모든 기능을 구현할 수 없음 |
| | **벤**더 종속성 탈피 | 특정 클라우드 벤더에 종속되지 않고 최고의 조합으로 인프라 구성 |
| | **F**ail-over | 높은 확장성을 기반 데이터, 워크플로우, 시스템을 백업 가능 |
| | **서**비스 탄력성 확보 | 서비스 니즈에 가장 적합한 컴퓨팅 자원 선정, 활용 |
| **공급자 측면** | Cloud MSP 기회 | 복수의 클라우드 매니지드 서비스의 새로운 기회 |
| | 통신/IDC 사업자 기회 | 글로벌 클라우드 사업자와 제휴 통해 클라우드 허브 역할 |

### 📊 멀티클라우드 vs 하이브리드 클라우드 비교

| 구분 | 멀티클라우드 | 하이브리드 클라우드 |
|:----|:-----------|:-----------------|
| **개념** | 레거시 DC + 2개 이상 CSP 조합 | 레거시 DC + 일부 클라우드 서비스 결합 |
| **접근방식** | 동일 유형 클라우드 2개 이상 | 퍼블릭 + 프라이빗 통합/오케스트레이션 |
| **클라우드 유형** | 퍼블릭 + 퍼블릭 | 퍼블릭 + 프라이빗 |
| **장점** | 다양한 CSP 서비스 활용 | 공유 코어 SW 서비스, 마이그레이션 유연성 |
| **목적** | Best-of-Breed, 벤더 종속 방지 | 보안과 유연성의 균형 |

### ⭐ 차별점 키워드

### 🏆 Cloud Brokerage (클라우드 브로커리지)
> 멀티클라우드 환경에서 **중개자 역할**을 수행하며, 사용자의 요구사항에 맞는 **최적의 클라우드 서비스를 선정/조합/관리**하는 서비스 모델
> - **통합(Aggregation)**: 여러 CSP 서비스를 단일 창구로 제공
> - **중개(Intermediation)**: 부가가치 서비스 추가 (보안, 모니터링)
> - **차익거래(Arbitrage)**: 비용 최적화를 위한 CSP 선택
