### 🔑 핵심 키워드 3개

| 키워드 | 쉬운 설명 |
|:------|:---------|
| **자원(Resource)** | URL로 접근하는 데이터 (사용자, 게시글 등) |
| **HTTP 메서드** | GET(조회), POST(생성), PUT(수정), DELETE(삭제) |
| **무상태(Stateless)** | 서버가 이전 요청을 기억 안 함 |

---

### 📖 등장배경

> **문제**: 예전에는 웹 서비스마다 통신 방식이 달랐어요...
> - 🔧 SOAP이 너무 복잡하고 무거움
> - 📦 XML만 써야 해서 데이터가 큼
> - 🤯 개발자마다 API 만드는 방식이 제각각
>
> **해결**: REST API 등장!
> - 📍 URL로 자원을 표현 (예: `/users/123`)
> - 🎯 HTTP 메서드로 행위 표현 (GET, POST, PUT, DELETE)
> - 📄 JSON으로 가볍게 데이터 전송

---

### 📝 개념 정의

> 🍔 **햄버거 가게 주문**에 비유하면...
>
> | REST 요소 | 햄버거 가게 비유 |
> |:---------|:---------------|
> | **URI (자원)** | 메뉴판의 햄버거 번호 (`/burger/1`) |
> | **GET** | "1번 햄버거 보여주세요" (조회) |
> | **POST** | "새 햄버거 만들어주세요" (생성) |
> | **PUT** | "1번 햄버거 치즈 추가해주세요" (수정) |
> | **DELETE** | "1번 햄버거 취소해주세요" (삭제) |
>
> **핵심**: **URL로 자원을 지정**하고, **HTTP 메서드로 행위를 표현**!

---

### 🏗️ 기술요소 (2그룹)

#### 그룹 1: REST 구성요소

| 요소 | 역할 |
|:----|:----|
| **URI** | 자원의 위치/이름 (예: `/users/123`) |
| **HTTP Method** | 행위 표현 (GET/POST/PUT/DELETE) |

#### 그룹 2: 데이터 형식

| 요소 | 역할 |
|:----|:----|
| **JSON** | 가볍고 읽기 쉬운 데이터 형식 |
| **HTTP 상태코드** | 결과 알림 (200 성공, 404 없음, 500 오류) |

---

### ⭐ 차별점 키워드

**Richardson Maturity Model (성숙도 모델)** 📊
> REST API가 얼마나 잘 설계됐는지 0~3단계로 평가
> - Level 0: HTTP만 사용 (RPC 스타일)
> - Level 1: URI로 리소스 구분
> - Level 2: HTTP 메서드 활용
> - Level 3: **HATEOAS** (하이퍼미디어 링크 제공)

## 🎯 기술사 수준 설명

### 🔑 핵심 키워드 3개

| 키워드 | 설명 |
|:------|:-----|
| **Stateless** | 서버가 클라이언트 상태를 저장하지 않음 (확장성 확보) |
| **Uniform Interface** | 일관된 인터페이스로 자원 접근 (URI + HTTP Method) |
| **Resource-Oriented** | 모든 것을 자원(Resource)으로 표현, URI로 식별 |

---

### 📖 등장배경 (2개)

| 배경 | 설명 |
|:----|:----|
| **SOAP의 복잡성** | XML 기반 SOAP의 무거움과 복잡한 스펙 (WSDL, UDDI) |
| **웹 표준 활용** | 기존 HTTP/URI 인프라를 그대로 활용한 경량 아키텍처 필요 |

## 🏗️ REST 구성요소 (2그룹 × 4항목) `자행표상`

### 그룹 1: REST 3대 구성요소 `자행표`

| 구성요소 | 설명 | 예시 |
|:--------|:-----|:----|
| **자**원 (Resource) | URI로 식별되는 모든 것 | `/users/123`, `/orders/456` |
| **행**위 (Verb) | HTTP Method로 표현 | GET, POST, PUT, DELETE |
| **표**현 (Representation) | 자원의 상태를 나타내는 데이터 | JSON, XML |

### 그룹 2: HTTP Method 4종 `CRUD`

| Method | 역할 | 멱등성 | 안전성 |
|:-------|:----|:-----:|:-----:|
| **GET** | 자원 조회 (Read) | ✅ | ✅ |
| **POST** | 자원 생성 (Create) | ❌ | ❌ |
| **PUT** | 자원 전체 수정 (Update) | ✅ | ❌ |
| **DELETE** | 자원 삭제 (Delete) | ✅ | ❌ |

---

### 🧠 암기법 상세

```
🧠 자행표 = "REST 3대 구성요소"

  자 : 자원 (Resource) - URI로 식별
  행 : 행위 (Verb) - HTTP Method
  표 : 표현 (Representation) - JSON/XML

🧠 CRUD = "HTTP Method 4종"

  C : Create → POST
  R : Read → GET
  U : Update → PUT (전체), PATCH (일부)
  D : Delete → DELETE
```

## 📊 REST 6대 제약조건 `클무캐유계코`

| 제약조건 | 설명 |
|:--------|:----|
| **클**라이언트-서버 (Client-Server) | 관심사 분리, 클라이언트와 서버 독립적 진화 |
| **무**상태 (Stateless) | 서버가 클라이언트 상태 저장 안 함, 요청에 필요한 모든 정보 포함 |
| **캐**시 가능 (Cacheable) | 응답 데이터 캐시 가능, 성능 향상 |
| **유**니폼 인터페이스 (Uniform Interface) | 일관된 인터페이스로 자원 접근 |
| **계**층화 시스템 (Layered System) | 중간 서버(프록시, 게이트웨이) 투명하게 배치 가능 |
| **코**드 온 디맨드 (Code on Demand) | 선택적, 서버에서 클라이언트로 실행 코드 전송 (JS 등) |

---

### 🧠 6대 제약조건 암기법

```
🧠 클무캐유계코 = "REST 6대 제약조건"

  클 : 클라이언트-서버 (관심사 분리)
  무 : 무상태 (Stateless, 확장성)
  캐 : 캐시 가능 (성능 향상)
  유 : 유니폼 인터페이스 (일관성)
  계 : 계층화 시스템 (유연성)
  코 : 코드 온 디맨드 (선택적)
```

## 📊 REST vs SOAP 비교

| 비교 항목 | REST | SOAP |
|:---------|:-----|:-----|
| **프로토콜** | HTTP만 사용 | HTTP, SMTP, TCP 등 |
| **데이터 형식** | JSON, XML, HTML 등 | XML만 사용 |
| **복잡성** | 단순, 경량 | 복잡, 무거움 |
| **상태 관리** | Stateless | Stateful 가능 |
| **보안** | HTTPS, OAuth | WS-Security |
| **캐싱** | HTTP 캐시 활용 | 불가능 |
| **표준 스펙** | 아키텍처 스타일 | WSDL, UDDI 등 |
| **사용 사례** | 웹/모바일 API | 엔터프라이즈 통합 |

## 📊 HTTP 상태코드 `2성3이4클5서`

| 코드 | 분류 | 주요 코드 |
|:----|:----|:---------|
| **2xx** | **성**공 | 200 OK, 201 Created, 204 No Content |
| **3xx** | **이**동 (Redirection) | 301 Moved Permanently, 304 Not Modified |
| **4xx** | **클**라이언트 오류 | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| **5xx** | **서**버 오류 | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

## 📊 RESTful API 설계 모범 사례

| 항목 | 좋은 예 | 나쁜 예 |
|:----|:-------|:-------|
| **명사형 URI** | `/users/123` | `/getUser?id=123` |
| **복수형 사용** | `/users` | `/user` |
| **계층 구조** | `/users/123/orders` | `/getUserOrders?userId=123` |
| **소문자 사용** | `/user-profiles` | `/UserProfiles` |
| **버전 관리** | `/v1/users` | `/users?version=1` |

## ⭐ 차별점 키워드

### 🏆 HATEOAS (Hypermedia As The Engine Of Application State)
> **하이퍼미디어를 애플리케이션 상태 엔진으로 사용**
> - 응답에 다음 행동을 위한 링크 포함
> - 클라이언트가 서버 구조를 몰라도 탐색 가능
> - Richardson Maturity Model **Level 3**

```json
{
  "id": 123,
  "name": "홍길동",
  "_links": {
    "self": { "href": "/users/123" },
    "orders": { "href": "/users/123/orders" },
    "delete": { "href": "/users/123", "method": "DELETE" }
  }
}
```
