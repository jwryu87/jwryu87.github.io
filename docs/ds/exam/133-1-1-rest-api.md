### 🔑 핵심 키워드 3개

| 키워드 | 쉬운 설명 |
|:------|:---------|
| **URI** | 자원의 주소 (예: `/users/홍길동`) |
| **HTTP Method** | GET(조회), POST(생성), PUT(수정), DELETE(삭제) |
| **무상태(Stateless)** | 서버가 이전 요청을 기억 안 함 |

---

### 📖 등장배경

> **문제**: 예전에는 웹 서비스마다 통신 방식이 달랐어요...
> - 🔧 SOAP이 너무 복잡하고 무거움
> - 📦 XML만 써야 해서 데이터가 큼
> - 🤯 개발자마다 API 만드는 방식이 제각각
>
> **해결**: REST API 등장!
> - 📍 URL로 자원을 표현 (예: `/users/itpe`)
> - 🎯 HTTP 메서드로 행위 표현 (GET, POST, PUT, DELETE)
> - 📄 JSON/XML로 데이터 전송

---

### 📝 개념 정의

> 🍔 **햄버거 가게 주문**에 비유하면...
>
> | REST 요소 | 햄버거 가게 비유 |
> |:---------|:---------------|
> | **URI (자원)** | 메뉴판의 햄버거 번호 (`/burger/1`) |
> | **GET** | "1번 햄버거 보여주세요" (조회) |
> | **POST** | "새 햄버거 만들어주세요" (생성) |
> | **PUT** | "1번 햄버거 치즈 추가해주세요" (수정) |
> | **DELETE** | "1번 햄버거 취소해주세요" (삭제) |
>
> **핵심**: **URL로 자원을 지정**하고, **HTTP 메서드로 행위를 표현**!

### ⭐ 차별점 키워드

**RESTful** 🎯
> REST 원칙을 **충실히 따르는** 서비스나 애플리케이션
> - REST: 이론적인 아키텍처 스타일 (원칙과 가이드라인)
> - RESTful: 실제로 REST를 구현한 것

## 📊 REST API의 특징 `클무캐계`

| 특징 | 설명 |
|:----|:----|
| **클**라이언트-서버 구조 | 클라이언트와 서버가 명확하게 분리되어 있어 서로의 **독립성을 보장** |
| **무**상태성 (Stateless) | 각 요청은 독립적이며 서버는 클라이언트의 **상태 정보를 저장하지 않음** |
| **캐**시 가능 (Cacheable) | HTTP의 특징인 **캐시 기능을 적용**하여, 응답시간 향상 |
| **계**층화된 시스템 | Client는 REST API Server만 호출하고, REST Server는 **다층 계층으로 구성** |

## 🏗️ REST 구성요소 (2그룹) `자행메`

### 그룹 1: REST 3대 구성요소 `자행메`

| 구성요소 | 설명 |
|:--------|:-----|
| **자**원 (Resource) | URI를 통해 식별 가능한 데이터 또는 서비스, Open API로 제공하는 대상 |
| **행**위 (Behavior) | 자원에 대한 생성·조회·수정·삭제 행위를 HTTP Method(POST, GET, PUT, DELETE)로 표현 |
| **메**시지 (Message) | 자원에 대한 행위를 위한 **요청정보**와 행위에 대한 결과인 **응답정보**를 출력하는 방식 |

### 그룹 2: HTTP Method 4종 `GPUD`

| Method | 역할 | URI 예시 |
|:-------|:----|:--------|
| **G**ET (조회) | 기존 자원에 대한 정보 조회 | `GET https://www.apisample.com/users` |
| **P**OST (생성) | 신규 자원의 생성 | `POST https://www.apisample.com/users/itpe` |
| **U**PUT (수정) | 기존 자원에 대한 정보 수정 | `PUT https://www.apisample.com/users/itpe` |
| **D**ELETE (삭제) | 기존 자원의 삭제 | `DELETE https://www.apisample.com/users/itpe` |

---

### 🧠 암기법 상세

```
🧠 자행메 = "REST 3대 구성요소"

  자 : 자원 (Resource) - URI로 식별
  행 : 행위 (Behavior) - HTTP Method
  메 : 메시지 (Message) - 요청/응답 정보

🧠 클무캐계 = "REST API 4대 특징"

  클 : 클라이언트-서버 구조 (독립성)
  무 : 무상태성 (Stateless)
  캐 : 캐시 가능 (응답시간 향상)
  계 : 계층화된 시스템 (다층 구조)
```

## 📊 REST vs RESTful 차이점

| 구분 | REST | RESTful |
|:----|:----|:-------|
| **정의** | 개념적 또는 이론적인 **아키텍처 스타일** | 실제 구현을 나타내며, REST 원칙을 **충실히 따르는** 서비스/앱 |
| **역할** | 웹 서비스를 설계하는 데 사용되는 **원칙과 가이드라인** 제공 | REST 아키텍처 스타일을 **구현**하고 있음을 의미 |
| **비유** | 설계도 (Blueprint) | 완성된 건물 (Building) |

> 💡 어떤 웹 서비스가 "**RESTful**"하다는 것은 그 서비스가 REST 아키텍처 스타일을 구현하고 있음을 의미

## 📊 REST 6대 제약조건 (심화) `클무캐유계코`

| 제약조건 | 설명 |
|:--------|:----|
| **클**라이언트-서버 (Client-Server) | 관심사 분리, 클라이언트와 서버 독립적 진화 |
| **무**상태 (Stateless) | 서버가 클라이언트 상태 저장 안 함, 요청에 필요한 모든 정보 포함 |
| **캐**시 가능 (Cacheable) | 응답 데이터 캐시 가능, 성능 향상 |
| **유**니폼 인터페이스 (Uniform Interface) | 일관된 인터페이스로 자원 접근 |
| **계**층화 시스템 (Layered System) | 중간 서버(프록시, 게이트웨이) 투명하게 배치 가능 |
| **코**드 온 디맨드 (Code on Demand) | 선택적, 서버에서 클라이언트로 실행 코드 전송 (JS 등) |

## 📊 REST vs SOAP 비교

| 비교 항목 | REST | SOAP |
|:---------|:-----|:-----|
| **프로토콜** | HTTP만 사용 | HTTP, SMTP, TCP 등 |
| **데이터 형식** | JSON, XML, HTML 등 | XML만 사용 |
| **복잡성** | 단순, 경량 | 복잡, 무거움 |
| **상태 관리** | Stateless | Stateful 가능 |
| **보안** | HTTPS, OAuth | WS-Security |
| **캐싱** | HTTP 캐시 활용 | 불가능 |
| **표준 스펙** | 아키텍처 스타일 | WSDL, UDDI 등 |
| **사용 사례** | 웹/모바일 API | 엔터프라이즈 통합 |

## 📊 HTTP 상태코드 `2성3이4클5서`

| 코드 | 분류 | 주요 코드 |
|:----|:----|:---------|
| **2xx** | **성**공 | 200 OK, 201 Created, 204 No Content |
| **3xx** | **이**동 (Redirection) | 301 Moved Permanently, 304 Not Modified |
| **4xx** | **클**라이언트 오류 | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found |
| **5xx** | **서**버 오류 | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable |

## 📊 Richardson Maturity Model (성숙도 모델)

```
┌─────────────────────────────────────────────────────────────────┐
│               Richardson Maturity Model                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Level 3: HATEOAS                           ← Glory of REST     │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ 하이퍼미디어 링크 제공 (다음 행동 안내)                  │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              ▲                                  │
│  Level 2: HTTP Verbs                                            │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ HTTP Method 활용 (GET, POST, PUT, DELETE)               │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              ▲                                  │
│  Level 1: Resources                                             │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ URI로 리소스 구분 (/users, /orders)                     │    │
│  └─────────────────────────────────────────────────────────┘    │
│                              ▲                                  │
│  Level 0: POX (Plain Old XML)               ← The Swamp of POX │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ HTTP를 터널로만 사용 (RPC 스타일)                       │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 📊 RESTful API 설계 모범 사례

| 항목 | 좋은 예 | 나쁜 예 |
|:----|:-------|:-------|
| **명사형 URI** | `/users/123` | `/getUser?id=123` |

| **버전 관리** | `/v1/users` | `/users?version=1` |

## ⭐ 차별점 키워드

### 🏆 HATEOAS (Hypermedia As The Engine Of Application State)
> **하이퍼미디어를 애플리케이션 상태 엔진으로 사용**
> - 응답에 다음 행동을 위한 링크 포함
> - 클라이언트가 서버 구조를 몰라도 탐색 가능
> - Richardson Maturity Model **Level 3**

```json
{
  "id": 123,
  "name": "홍길동",
  "_links": {
    "self": { "href": "/users/123" },
    "orders": { "href": "/users/123/orders" },
    "delete": { "href": "/users/123", "method": "DELETE" }
  }
}
```
