### 📖 등장배경

> **문제**: 컨테이너가 많아지니까...
> - 📦 도커 컨테이너 수백 개를 어떻게 관리하지?
> - 🔄 컨테이너가 죽으면 누가 다시 살려?
> - ⚖️ 트래픽이 몰리면 자동으로 늘려줘야 하는데...
>
> **해결**: 쿠버네티스 등장!
> - 🎭 컨테이너 자동 배치, 복제, 확장
> - 🔧 장애 발생 시 자동 복구
> - ⚡ 부하에 따라 자동 스케일링

---

### 📝 개념 정의

> 🎼 **오케스트라 지휘자**에 비유하면...
>
> | 쿠버네티스 | 오케스트라 비유 |
> |:---------|:--------------|
> | **Master Node** | 지휘자 (전체 조율) |
> | **Worker Node** | 연주자들 (실제 연주) |
> | **Pod** | 악기 한 대 (연주 단위) |
> | **kubectl** | 지휘봉 (명령 도구) |
>
> **핵심**: 컨테이너를 **자동으로 배치, 관리, 확장**하는 플랫폼!

---

### 🏗️ 기술요소 (2그룹)

#### 그룹 1: Master Node (Control Plane)

| 요소 | 역할 |
|:----|:----|
| **API Server** | 명령을 전달하는 창구 |
| **Scheduler** | Pod를 어느 노드에 배치할지 결정 |
| **Controller Manager** | Pod 생성, 복제, 장애 조치 |
| **etcd** | 클러스터 데이터 저장소 |

#### 그룹 2: Worker Node

| 요소 | 역할 |
|:----|:----|
| **Pod** | 컨테이너를 담는 최소 배포 단위 |
| **Kubelet** | Node의 에이전트 (명령 수행) |
| **Kube-Proxy** | 네트워크, 로드밸런싱 |
| **Container Runtime** | 컨테이너 실행 (Docker 등) |

---

### ⭐ 차별점 키워드

**Self-Healing (자가 복구)** 🔄
> 컨테이너가 죽으면 **자동으로 재시작**하고
> 노드가 죽으면 **다른 노드에 Pod를 재배치**

## 📊 쿠버네티스 주요 기능 `복빈앱스확복`

| 기능 | 설명 |
|:----|:----|
| **복**수 호스트 관리 | 여러 호스트에 걸쳐 컨테이너를 오케스트레이션 관리 |
| **빈** 패킹 (Bin Packing) | 컨테이너별 필요한 CPU 및 메모리의 크기를 지정 |
| **앱**플리케이션 자동화 | 애플리케이션 배포 및 업데이트를 제어하고 자동화 |
| **스**테이트풀 (Stateful) | 스토리지를 장착 및 추가해 Stateful 애플리케이션 실행 |
| **확**장성 | 컨테이너화 된 애플리케이션과 해당 리소스를 즉시 확장 가능 |
| **복**구 자동화 | 자동 배치, 재시작, 복제, 확장을 사용해 셀프 복구 |

## 🏗️ 구성요소 (2그룹) `A-S-C-E` `P-K-K-C`

### 그룹 1: Master Node (Control Plane) `A-S-C-E`

| 구성요소 | 설명 |
|:--------|:----|
| **A**PI Server | 간단하게 명령어를 전달해주는 역할, etcd 클러스터와 REST방식 통신, kubectl 명령어를 통해 받은 작업을 API서버로 전송 |
| **S**cheduler | Pod, 서비스 등 각 자원을 적절한 노드에 할당, 공유 상태 스케줄링(Shared-State Scheduling)으로 리소스 할당 결정 |
| **C**ontroller Manager | Replication(Pod 생성), Services(분산 요청 관리), Node(장애 조치), End-Point(서비스-Pod 연결) Controller |
| **e**tcd | 모든 클러스터 데이터를 담는 분산 저장소 (Key-Value Store) |

### 그룹 2: Worker Node `P-K-K-C`

| 구성요소 | 설명 |
|:--------|:----|
| **P**od | 쿠버네티스에서 가장 기본적인 배포 단위로, 컨테이너를 포함하는 단위 |
| **K**ubelet | Node에 배포되는 에이전트, Pod에서 컨테이너 동작을 관리, Master의 API 서버를 통해 명령 수행 및 상태 정보 전달 |
| **K**ube-Proxy | 컨테이너(Pod)간 네트워크 프록시 및 Load Balancing 수행, 가상 네트워크 상에서 트래픽 라우팅 및 Pod IP 관리 |
| **C**ontainer Runtime | Pod를 통해서 배포된 컨테이너를 실행 (Docker, containerd, CRI-O 등) |

---

### 🧠 암기법 상세

```
🧠 A-S-C-E = "Master Node 4대 컴포넌트"

  A : API Server (명령 전달, REST 통신)
  S : Scheduler (Pod 배치 결정)
  C : Controller Manager (Pod 생성/복제/장애조치)
  E : etcd (클러스터 데이터 저장)

🧠 P-K-K-C = "Worker Node 4대 컴포넌트"

  P : Pod (컨테이너 배포 단위)
  K : Kubelet (Node 에이전트)
  K : Kube-Proxy (네트워크, 로드밸런싱)
  C : Container Runtime (Docker 등)
```

## 📊 Controller Manager 상세 `R-S-N-E`

| Controller | 역할 |
|:-----------|:----|
| **R**eplication Controller | Pod 기동 및 장애 발생시 새로운 Pod 생성 |
| **S**ervices Controller | 신규로 생성된 Pod의 분하 분산 요청 관리 |
| **N**ode Controller | Node 장애 발생시 알림과 장애 조치 역할 수행 |
| **E**nd-Point Controller | 서비스와 Pod를 연결 |

## ⭐ 차별점 키워드

### 🏆 Shared-State Scheduling (공유 상태 스케줄링)
> Scheduler가 **클러스터 전체의 상태 정보**를 공유하여 최적의 노드에 Pod를 배치하는 방식
> - 모든 스케줄러가 동일한 상태 정보 참조
> - 낙관적 동시성 제어 (Optimistic Concurrency Control) 사용
> - 충돌 시 재시도로 일관성 유지
