### 📖 등장배경

> **문제**: 현금 사용이 줄고 디지털 결제가 늘어나는데...
> - 💳 민간 디지털 화폐(페이, 암호화폐)만 있으면 안전할까?
> - 🏦 중앙은행이 직접 관리하는 디지털 화폐가 필요!
>
> **해결**: CBDC 등장!
> - 현금처럼 중앙은행이 발행하지만 디지털 형태
> - 비트코인과 달리 국가가 가치를 보증

---

### 📝 개념 정의

> 💰 **지폐 vs CBDC**에 비유하면...
>
> | 구분 | 현금(지폐) | CBDC |
> |:----|:----------|:----|
> | **형태** | 종이 | 디지털 |
> | **발행자** | 중앙은행 | 중앙은행 |
> | **사용** | 물리적 교환 | 앱/전자지갑 |
> | **추적** | 어려움 | 가능 |

---

### 🏗️ 설계 시 고려할 4가지 관점

| 관점 | 핵심 질문 |
|:----|:---------|
| **아키텍처** | 누가 발행하고 누가 배포할까? |
| **인프라** | 중앙집중? 분산원장(블록체인)? |
| **접근** | 계좌 방식? 토큰 방식? |
| **연계** | 다른 나라와 어떻게 연결할까? |

---

### ⭐ 차별점 키워드

**프로그래머블 화폐 (Programmable Money)** 💻
> 스마트 컨트랙트처럼 **조건부 결제**가 가능한 화폐
> - 예: "물건 받으면 자동 결제"
> - 에스크로 없이 거래 완결 가능

## 📝 II. CBDC 설계를 위한 아키텍처 및 인프라 관점 `청중관적선`

### 가. 아키텍처 관점 고려사항 `청중`

#### 1. 청구권 설계 (직접/간접)

| 유형 | 설명 |
|:----|:----|
| **직접 청구권** | 소비자가 중앙은행에 직접 계좌 보유, 중앙은행이 거래 정보 직접 처리 |
| **간접 청구권** | 중개기관이 소비자에게 1:1 보증된 간접 CBDC 발행 (IMF 모델) |
| **절충형** | 소비자의 중앙은행 청구권 유지 + 소매업무는 중계기관에 이양 |

#### 2. 중계기관 역할 설계 `Two-Plat-Syn`

| 유형 | 설명 |
|:----|:----|
| **Two**-Tiered Issuance | 규제된 중개자에게 발행되어 배포되는 모델 |
| **Plat**form Approach | 민간 기업이 고객 대면 서비스 제공, CBDC 기술 플랫폼 구축 |
| **Syn**thetic Issuers | Narrow banking과 유사한 발행사가 발행하는 CBDC |

---

### 나. 인프라 관점 고려사항 `관적선`

| 고려사항 | 핵심 내용 |
|:--------|:---------|
| **관**리 (신뢰 메커니즘) | 중앙집중식 vs DLT(분산원장) 선택 |
| **적**합성 | 아키텍처 유형에 따른 인프라 요구사항 매칭 |
| **선**호도 | 과도한 설계 여부 평가 (중앙은행 신뢰도 감안) |

#### DLT 적용 시 고려사항

```
┌─────────────────────────────────────────────────────────────────┐
│                  DLT vs 중앙집중식 비교                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌─────────────────────┐     ┌─────────────────────┐           │
│   │    중앙집중식       │     │   DLT (분산원장)    │           │
│   │  Centralized DB     │     │ Distributed Ledger  │           │
│   ├─────────────────────┤     ├─────────────────────┤           │

│   └─────────────────────┘     └─────────────────────┘           │
│                                                                  │
│   💡 현재 판단: DLT는 소규모 도매 CBDC에 적합                    │
│      소매 CBDC는 현실적으로 중앙집중식이 적합                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 📝 III. CBDC 설계를 위한 접근 및 연계 관점 `방보개국`

### 가. 접근 관점 고려사항 `방보개`

| 고려사항 | 핵심 내용 |
|:--------|:---------|
| **방**식 | 계좌 기반 vs 토큰 기반 선택 |
| **보**안 | 영지식증명 등 암호화폐 기술 적용한 프라이버시 |
| **개**발 | 스마트 컨트랙트 활용 프로그래머블 화폐 |

#### 계좌 방식 vs 토큰 방식

| 구분 | 계좌 방식 (Account-based) | 토큰 방식 (Token-based) |
|:----|:------------------------|:----------------------|
| **신원 확인** | 청구권자 신원(identity)으로 기록 | 암호화된 디지털 사인으로 증명 |
| **중개기관** | 필요 | 불필요 (P2P 거래) |
| **사례** | 은행 계좌, 모바일페이 | 암호화폐, 현금과 유사 |
| **국경간 연계** | 국제간 공조 필요 | 해외 직접 접근 가능 |

#### 프로그래머블 화폐 (스마트 컨트랙트 활용)

```
┌─────────────────────────────────────────────────────────────────┐
│              프로그래머블 화폐 활용 예시                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   기존 방식 (에스크로 필요)                                      │
│   ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐                  │
│   │ 구매자│───►│에스크로│───►│ 판매자│───►│ 물건 │                  │
│   └──────┘    └──────┘    └──────┘    └──────┘                  │
│                  ↑                                              │
│            제3자 중개자 필요                                     │
│                                                                  │
│   프로그래머블 화폐 (Atomic Settlement)                          │
│   ┌──────┐              ┌──────┐                                │
│   │ 구매자│◄────────────►│ 판매자│                                │
│   └──────┘    조건 충족   └──────┘                                │
│               시 자동 결제                                       │
│               (중개자 불필요)                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 나. 연계 관점 고려사항 `국`

| 고려사항 | 핵심 내용 |
|:--------|:---------|
| **국**가 연계 | 국경간 결제 효율화, mCBDC(다자간 CBDC) 고려 |

#### 국경간 CBDC 연계 유형

| 유형 | 설명 |
|:----|:----|
| **도매간 상호연계** | Wholesale linkage, 기존 방식 개선 |
| **소매간 상호연계** | Retail linkage, 소비자가 여러 나라 디지털 통화 보유 |
| **mCBDC** | Multi-CBDC, 다자간 CBDC 상호운용 |

#### 국가별 CBDC 추진 현황

| 국가 | 추진 단계 |
|:----|:---------|
| 🇨🇳 **중국** | 적극적 도입 진행 중 (디지털 위안) |
| 🇸🇪 **스웨덴** | 본격적 의사결정 단계 (e-Krona) |
| 🇬🇧 **영국** | 본격적 의사결정 단계 |
| 🇺🇸 **미국** | 신중한 접근, 검토 단계 |
| 🇰🇷 **한국** | 파일럿 테스트 진행 |

## ⭐ 차별점 키워드

### 🏆 프로그래머블 화폐 (Programmable Money)
> 스마트 컨트랙트 기술을 활용하여 **조건부 결제**가 가능한 화폐
> - **Atomic Settlement**: 중개자 없이 조건 충족 시 자동 결제
> - **활용 사례**: 전자상거래, 기관간 거래, 소액결제
> - **기술 기반**: 암호화폐의 스마트 컨트랙트 기술 적용

---

### 📋 6하원칙 요약

| 구분 | 내용 |
|:----|:----|
| **누가 (Who)** | 중앙은행 (한국은행, Fed, ECB, PBoC 등) |
| **무엇을 (What)** | 전자적 형태의 법정화폐 발행 |
| **언제 (When)** | 현금 사용 감소, 디지털 결제 확산 시대 |
| **어디서 (Where)** | 전 세계 중앙은행에서 연구/파일럿 진행 |
| **왜 (Why)** | 금융 안정성, 결제 효율성, 금융 포용성 확보 |
| **어떻게 (How)** | 아키텍처/인프라/접근/연계 4대 관점 설계 |
