# Scale-up / Scale-out
{: .fs-8 }

18. DS 기타
{: .label .label-purple }

---

## 핵심 키워드

`수직확장` `수평확장` `성능 향상` `처리 능력`

---

## 정의/개념

| 구분 | 정의 |
|:-----|:-----|
| **Scale-up** | **서버 그 자체를 증강**하는 것에 의해서 처리 능력을 향상시키는 **기술** |
| **Scale-out** | **접속된 서버의 대수를 늘려** 처리 능력을 향상시키는 **기술** |

---

## Scale-up vs Scale-out 비교

| 구분 | Scale-Up (수직 확장) | Scale-Out (수평 확장) |
|:-----|:-----|:-----|
| **구성도** | 서버 1대 → 고사양 서버 1대 | 서버 1대 → 서버 여러 대 |
| **구조** | 더 빠른 속도의 CPU 로 변경하거나, 더 많은 RAM 을 추가하는 등의 하드웨어 장비의 성능을 높이는 방식 | 하나의 장비에서 처리하던 일을 여러 장비에 나눠서 처리할 수 있도록 설계 변경 |
| **확장** | 수직 확장, 성능 확장에 한계가 있음 | 수평 확장, 지속적 확장이 가능 |
| **장애** | 한 대의 서버에 부하가 집중되므로 장애 시 영향도가 큼 | 읽기/쓰기가 여러 대의 서버에 분산되어 처리됨으로 장애 시 전면 장애의 가능성이 적음 |
| **주요기술** | 고성능 CPU, Memory 확장, SSD | Sharding, Query-off Loading, Queue, In-Memory Cache, NoSQL |
| **주요용도** | - 고성능 Legacy 애플리케이션 구축<br>- 구축이 쉽고 관리 용이 | - 분산처리 시스템/Global 웹 애플리케이션<br>- 점진적 증가 가능 |

---

## Scale-up 상세

### 특징

| 특징 | 설명 |
|:-----|:-----|
| **방식** | 단일 서버의 하드웨어 성능 향상 |
| **장점** | 구축이 쉽고 관리가 용이함 |
| **단점** | 성능 확장에 물리적 한계 존재 |
| **적용 사례** | 고성능 데이터베이스, Legacy 시스템 |

### 주요 기술

- 고성능 CPU 업그레이드
- Memory(RAM) 용량 확장
- SSD 스토리지 적용
- 네트워크 카드 성능 향상

---

## Scale-out 상세

### 특징

| 특징 | 설명 |
|:-----|:-----|
| **방식** | 서버 수를 늘려 부하 분산 |
| **장점** | 지속적인 확장 가능, 장애 복원력 높음 |
| **단점** | 분산 처리 설계 필요, 복잡도 증가 |
| **적용 사례** | 웹 서비스, 분산 시스템, 클라우드 |

### 주요 기술

| 기술 | 설명 |
|:-----|:-----|
| **Sharding** | 데이터를 여러 DB에 분산 저장 |
| **Query-off Loading** | 읽기 쿼리를 복제본으로 분산 |
| **Queue** | 비동기 메시지 처리 |
| **In-Memory Cache** | Redis, Memcached 등 메모리 캐싱 |
| **NoSQL** | MongoDB, Cassandra 등 분산 DB |
| **Load Balancer** | 트래픽 분산 처리 |

---

## 선택 기준

| 상황 | 권장 방식 |
|:-----|:-----|
| 빠른 구축 필요 | Scale-up |
| 예측 가능한 부하 증가 | Scale-out |
| 고가용성 필요 | Scale-out |
| Legacy 시스템 | Scale-up |
| 클라우드 환경 | Scale-out |
| 비용 효율성 중시 | Scale-out |

---

## 하이브리드 접근

실제 환경에서는 Scale-up과 Scale-out을 함께 사용하는 것이 일반적:

1. **초기**: Scale-up으로 빠른 성능 확보
2. **성장기**: Scale-out으로 확장성 확보
3. **안정기**: 두 방식 최적 조합으로 운영
