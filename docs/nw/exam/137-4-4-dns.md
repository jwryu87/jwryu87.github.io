### 한 줄 정의

> DNS는 **도메인 이름을 IP 주소로 변환해주는 인터넷 전화번호부**입니다.

### 쉬운 비유

> 친구 이름으로 전화번호를 찾아주는 **전화번호부**와 같습니다. "www.google.com"이라는 이름을 입력하면, DNS가 이를 "142.250.196.100"이라는 IP 주소로 변환해줍니다.

### 왜 필요한가?

1. **숫자 기억의 어려움**: IP 주소는 숫자로 이루어져 있어 사람이 기억하기 어렵습니다.
2. **도메인 이름의 편의성**: "www.google.com"처럼 의미 있는 이름으로 접근할 수 있습니다.
3. **분산 관리 체계**: 전 세계적으로 분산된 DNS 서버가 도메인 정보를 관리하여 효율성을 높입니다.

## 1. DNS 개요

{: .highlight }
> **DNS 개요**: IP 주소 변환, 도메인 관리, 분산 데이터베이스
> - (개념) `도메인↔IP 변환`
> - (주요기능) `네중도유`
> - (동작절차) `쿼리응답`

### 가. "IP 주소 변환", DNS의 개념

네트워크를 통해 원격의 컴퓨터에 접속하기 위해서는 IP 주소를 이용해야 하지만, 숫자의 연속인 **IP 주소를 일일이 외울 수 없기 때문**에 인간이 쉽게 기억할 수 있는 **도메인 주소 체계**가 만들어짐

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           DNS 동작 개념도                                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  ┌──────────────┐    ① 도메인 질의     ┌──────────────┐                │
│  │   Browser    │ ───────────────────> │  DNS Server  │                │
│  │              │                      │              │                │
│  │  www.itpe.com│ <─────────────────── │  Zone File   │                │
│  └──────┬───────┘    ② IP 주소 전달    │  Zone Data   │                │
│         │                              └──────────────┘                │
│         │                                                               │
│         │ ③ 웹 서버로 접속                                              │
│         ▼                                                               │
│  ┌──────────────┐                                                      │
│  │  Web Server  │                                                      │
│  │ (12.34.56.7) │                                                      │
│  └──────────────┘                                                      │
└─────────────────────────────────────────────────────────────────────────┘
```

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 도메인이나 호스트 이름을 컴퓨터가 처리하기 용이한 숫자 형태의 **IP 주소로 매핑 변환하는 시스템** |

| 역할 | 설명 |
|:----|:-----|
| 호스트 네임에 대한 NW 주소 변환 | URL을 IP Address로 변환 |
| 중복 없는 네임 관리체계 유지 | 도메인 네임 유일성 보장 |
| 도메인 네임의 생성/변경/삭제 독자적 수행 | 분산 관리 체계 |
| 유연성 및 확장을 위한 분산 데이터베이스 체계 | 계층적 구조 |

### 나. DNS의 주요기능 `네중도유`

| 기능 | 상세 내용 |
|:-----|:---------|
| **네**임 해석 (Name Resolution) | URL을 IP Address로 변환 |
| **중**복 없는 네임 관리 | 도메인 네임 유일성 보장 |
| **도**메인 독자적 관리 | 도메인 네임의 생성/변경/삭제 독자적 수행, 분산 관리 체계 |
| **유**연성 및 확장 | 분산 데이터베이스 체계, 계층적 구조 |

### 다. DNS의 동작 절차

| 구분 | 절차 | 설명 |
|:-----|:-----|:-----|
| **쿼리 단계** | 1) 도메인 네임 입력 | 사용자가 브라우저에 도메인 네임 입력, 쿼리 시작 |
| | 2) 호스트 캐시 확인 | OS 내 DNS 리졸버가 호스트 캐시(로컬 캐시) 확인 |
| **응답 단계** | 3) 루트 네임 서버 질의 | 캐시에 없으면 지정된 DNS 리졸버에 요청, 리졸버가 사용자를 대신해 최종 IP를 찾는 역할 수행 |
| | 4) TLD, 권한 서버 질의 | TLD(top-Level Domain, 최상위 도메인): 인터넷 도메인 네임 체계(DNS)에서 맨 오른쪽에 위치한 부분 (com, net) |
| | 5) IP 반환 및 캐싱 | IP를 통해 인터넷 접속, 로컬 캐싱 |

> DNS 절차를 수행하기 위한 호스트 및 네임 서버 구성 요소 필요

## 2. DNS 구성요소

{: .highlight }
> **DNS 구성요소**: 도메인 공간, 리소스 레코드, 존 파일, 네임서버, 리졸버
> - (구성요소) `도리존네리`
> - (서버계층) `루TLD권한`
> - (호스트측면) `OS데이터`
> - (네임서버측면) `제어데이터`

### 가. DNS 구성도

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│  ┌──────────────┐                   ┌─────────────────────────────────┐│
│  │  호스트(Host) │                   │       도메인 네임 공간            ││
│  │              │                   │    (Domain Name Space)          ││
│  │ Application  │                   │                                 ││
│  │      │       │                   │       리소스 레코드              ││
│  │  Socket API  │                   │    (Resource Records)           ││
│  │      │       │                   ├─────────────────────────────────┤│
│  │      ▼       │                   │                                 ││
│  │ ┌──────────┐ │                   │  존 파일(Zone File)             ││
│  │ │ Resolver │ │   DNS 질의/응답    │         │                       ││
│  │ │          │ │ <──────────────> │         ▼                       ││
│  │ │  Cache   │ │   UDP/TCP 53     │  ┌─────────────┐                ││
│  │ │  Data    │ │                   │  │ Name Server │                ││
│  │ └──────────┘ │                   │  │             │                ││
│  └──────────────┘                   │  │ Domain Zone │                ││
│                                     │  │    Data     │                ││
│                                     │  └─────────────┘                ││
│                                     └─────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────────────┘
```

### 나. DNS 구성요소 상세 `도리존네리`

#### 그룹 1: DNS 핵심 구성요소 `도리존네리`

| 구성요소 | 특징 | 설명 |
|:--------|:-----|:-----|
| **도**메인 네임 공간 (Domain Name Space) | 계층 구조 | 도메인 네임을 중복되지 않게 생성하여 사용하도록 정의된 도메인 네임 구성 체계, 한 개 이상의 레이블을 점으로 구분하여 사용, 계층적(트리) 구조, 유연성, 확장성 |
| **리**소스 레코드 (Resource Record) | 도메인 정보 제공 | 도메인에 대한 정보를 제공하고 도메인을 지원하는 다양한 데이터 유형들로 각 필드들은 스페이스(Space)로 구분 |
| **존** 파일 (Zone File) | SOA, 응답 레코드 구성 | SOA(Start of Authority) 영역과 응답 레코드 영역으로 구성되어 있으며 도메인을 IP 주소 또는 URL 등으로 변환해주는 역할 |
| **네**임 서버 (Name Server) | 도메인 데이터 보유 | 도메인 네임에 대한 데이터 질의가 있을 때, 보유하고 있는 도메인의 데이터를 조회하여 응답 |

#### 그룹 2: DNS 클라이언트 구성요소 `리졸버`

| 구성요소 | 특징 | 설명 |
|:--------|:-----|:-----|
| **리**졸버 (Resolver) | 정보 조회, 추출 | 네임 서버에 원하는 호스트에 대한 정보를 조회(질의) 및 추출하는 DNS 클라이언트용 프로그램 |
| **캐**시 리졸버 (Cache Resolver) | 로컬 캐시 관리 | OS가 관리하는 DNS 캐시, 호스트 캐시 확인 |
| **재**귀적 리졸버 (Recursive Resolver) | ISP/기업 리졸버 | ISP, 기업 네트워크 서버 내 리졸버, 사용자를 대신해 최종 IP를 찾는 역할 수행 |
| **스**텁 리졸버 (Stub Resolver) | 클라이언트 측 리졸버 | 애플리케이션 요청을 받아 DNS 서버에 질의하는 모듈, DNS 클라이언트 |

### 다. DNS 서버계층 구조 `루TLD권한`

#### 그룹 1: DNS 서버 계층 구조 `루TLD권한`

| 구성요소 | 설명 |
|:--------|:-----|
| **루**트 네임서버 | 도메인 체계의 최상위, TLD 서버 위치를 알려주는 서버 |
| **TLD** 네임서버 | 최상위 도메인 관리 네임서버 (.com, .net, .kr 등) |
| **권한** 네임서버 | 최종 응답 제공, 실제 도메인-IP 매핑 정보 보유 |
| **재**귀 네임서버 | 클라이언트의 질의를 대신 수행하여 최종 응답을 받아 전달하는 서버 |

#### 그룹 2: DNS 서버 유형 `주권재`

| 구성요소 | 설명 |
|:--------|:-----|
| **주** 네임서버 (Primary Name Server) | 존 파일의 원본을 관리하는 서버, 존 데이터의 변경 권한 보유 |
| **권**한 네임서버 (Authoritative Name Server) | 특정 도메인에 대한 최종 권한을 가진 서버, 실제 도메인-IP 매핑 정보 보유 |
| **재**귀 네임서버 (Recursive Name Server) | 클라이언트의 질의를 대신 수행하여 최종 응답을 받아 전달하는 서버 |
| **캐**시 네임서버 (Caching Name Server) | 이전 질의 결과를 저장하여 빠른 응답을 제공하는 서버 |

### 라. 호스트 측면 구성요소 `OS데이터`

#### 그룹 1: OS 측면 구성요소 `OS`

| 구성요소 | 설명 |
|:--------|:-----|

| **DNS API** | 브라우저, 외부 네임서버와의 인터페이스 제공 |
| **Socket API** | 네트워크 통신을 위한 소켓 인터페이스 |
| **Application** | DNS를 사용하는 애플리케이션 (브라우저, 메일 클라이언트 등) |

#### 그룹 2: 데이터 측면 구성요소 `데이터`

| 구성요소 | 설명 |
|:--------|:-----|
| **Host(local) DNS 캐시** | OS가 관리하는 DNS 캐시 |
| **Hosts 파일** | 로컬 수동 매핑 파일, hosts 파일에 사용자가 직접 도메인 이름과 IP 주소를 매핑하여 지정 |
| **브라우저 캐시** | 웹 브라우저(Chrome, Edge, Firefox 등)가 자체적으로 저장해 두는 도메인↔IP 주소 매핑 정보, OS나 DNS까지 거치지 않고 바로 사용 가능 |
| **Resolver Cache Data** | 리졸버가 저장하는 캐시 데이터 |

### 마. 네임서버 측면 구성요소 `제어데이터`

#### 그룹 1: 제어 측면 구성요소 `제어`

| 구성요소 | 설명 |
|:--------|:-----|
| **재**귀적 리졸버 | Recursive Resolver: ISP, 기업 네트워크 서버 내 리졸버 |
| **Zone File** | 특정 도메인 영역(zone)에 대한 DNS 레코드들을 모아놓은 텍스트 파일 |
| **도**메인 네임 공간 | 도메인 네임을 중복되지 않게 네임을 생성하여 사용하도록 정의된 도메인 네임 구성 체계 |
| **리**소스 레코드 | 도메인에 대한 정보를 제공하고 도메인을 지원하는 다양한 데이터 유형들로 각 필드들은 스페이스로 구분 |

#### 그룹 2: 데이터 측면 구성요소 `데이터`

| 구성요소 | 설명 |
|:--------|:-----|

## 3. DNS의 보안 취약점 및 대응 방안

{: .highlight }
> **DNS 보안**: 사용자 취약점, 운영 취약점, 대응 방안
> - (사용자 취약점) `악스개도`
> - (운영 취약점) `캐DDoS중서`
> - (대응방안) `DoH/DoT` `DNSSEC`

### 가. DNS의 보안 취약점

#### 그룹 1: 사용자 측면 보안 취약점 `악스개도`

| 보안 취약점 | 설명 |
|:-----------|:-----|
| **악**성 리다이렉트 | URL 유지 상태로 전혀 다른(피싱) 사이트로 연결 |
| DNS **스**푸핑 | DNS 응답 조작, 사용자를 가짜 목적지로 이동 시킴 |
| **개**인정보 유출/도용 | DNS 서버 경유 시 개인 정보 유출 가능 (클라이언트 IP 노출, 위치 정보 등) |
| **도**메인 하이재킹 | 도메인 자체 소유권, 설정 탈취 |

#### 그룹 2: 운영 측면 보안 취약점 `캐DDoS중서`

| 보안 취약점 | 설명 |
|:-----------|:-----|
| **캐**시 포이즈닝 | 캐시 내 데이터 오염 공격 |
| **DDoS** 공격 | DNS 네임 서버에 대한 서비스 증폭 공격, 서비스 불능 유도 |
| **중**간자 공격 | 호스트-네임서버, 네임서버-네임서버간 공유 정보 탈취 공격 |
| **서**버 권한 탈취 | 네임 서버, 권한 서버 권한 탈취, 조작 |

### 나. DNS의 보안 취약점 대응 방안

#### 그룹 1: 사용자 측면 대응 방안 `DoH/DoT`

| 대응방안 | 설명 |
|:--------|:-----|
| **DoH/DoT 활성화** | DoH(DNS over HTTPS), DoT(DNS over TLS)로 암호화된 DNS 설정 |
| **HTTPS 인증서 갱신** | 인증서 불일치 경고 확인, 갱신 |
| **VPN 사용** | 공용 망 내 DNS 요청 경로 VPN 통한 자체 보호 |
| **공용망/클릭 주의** | 공용 wifi 내 민감정보 이용 금지 |

#### 그룹 2: 운영 측면 대응 방안 `DNSSEC`

| 대응방안 | 설명 |
|:--------|:-----|
| **DNSSEC** | 응답 위·변조 방지(무결성 확보), 전자서명+HASH로 데이터 원본성/무결성 확인 |
| **DoH/DoT 운영** | DoH/DoT를 지원하는 리졸버 운영, 구간 암호화 |
| **응답률 제한** | DDoS 공격 예방 |
| **로그, 모니터링** | 비정상 트래픽·쿼리 패턴 감시, 알림 체계 운영 |

### 다. DNS 증폭 DDoS 공격 개념 및 대응 방안

| 구분 | 내용 |
|:-----|:-----|
| **개념** | DNS ANY Query를 요청하는 것으로 위장한 후, ANY 레코드의 결과를 중계 DNS 서버를 통해 공격 대상이 되는 DNS로 동시 전송(DDoS 증폭 쿼리)하여 성능저하를 초래한 공격 |
| **대응방안** | Recursion 서비스를 신뢰된 host로 제한, Recursion 기능 비활성화, Black hole 옵션 이용한 DNS 질의 제한 |

### 라. DNS Cache Poisoning Attack 개념 및 대응 방안

| 구분 | 내용 |
|:-----|:-----|
| **개념** | 취약한 DNS 서버에 조작된 Query를 전송하여 DNS 서버가 저장하고 있는 주소 Cache 정보를 임의로 변조하는 공격 |
| **대응방안** | DNSSEC의 설치 및 운영, Name Server의 recursive 기능 제한, DNS 관련 서버 보안 강화 |

### 마. DNS 보안취약점에 따른 추가 대응방안

| 관점 | 대응방안 | 설명 |
|:-----|:--------|:-----|
| **엔드 포인트** | 암호화 기반 DNS 통신 | 엔드 포인트와 캐시 DNS 사이 통신 암호화로 보안 강화 |
| | DoT (DNS Over TLS) | 전송계층 암호화 적용, TLS 프로토콜, 853 Port |
| | DoH (DNS Over HTTPS) | 응용계층 암호화 적용, HTTPS 암호통신, 443 Port |
| **서버 설정** | Recursion Query 제한 | 특정 Zone에 대한 요청에만 응답할 수 있도록 "allow-query" 설정 |
| | Zone Transfer 제한 | TSIG(Trusted Signature)를 이용한 암호키를 이용하여 Zone 데이터 인증 및 검증 |
| | Bind 업데이트 | 항상 최신 버전의 BIND를 사용하여 DNS 코드 취약점 제거 |
| **외부 시스템 활용** | DNS 싱크홀 | 악성봇에 감염된 PC를 해커가 조종하지 못하도록 악성봇과 해커의 명령/제어 서버 간 연결을 차단하는 서비스 |
| | DDoS 사이버대피소 | 피해 웹사이트로 향하는 공격 트래픽을 대피소로 우회하여 분석, 차단함으로써 정상적으로 운영 |

> DNS 보안 취약점은 파밍 공격, DNS 스푸핑 공격 등이 있으며, 근본적으로 보안 대응을 위해서는 DNS에 전송되는 데이터의 위·변조 여부를 검출하기 위해 전자서명 보안기술을 확장하여 적용한 **IETF DNS 보안표준(DNSSEC)을 적용 필요**
