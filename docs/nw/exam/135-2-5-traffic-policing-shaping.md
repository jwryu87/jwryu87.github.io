### 한 줄 정의
> **트래픽 폴리싱**: 초과 트래픽을 **즉시 버림(Drop)**으로써 대역폭을 제어하는 QoS 기술
> **트래픽 쉐이핑**: 초과 트래픽을 **버퍼에 저장 후 천천히 전송**하여 대역폭을 조절하는 QoS 기술

### 쉬운 비유 🚗
> **고속도로 요금소**와 같아요!
> - **폴리싱 (경찰)**: 🚔 과속 차량을 **즉시 단속**해서 통과 못하게 함 (Drop)
> - **쉐이핑 (신호등)**: 🚦 차량이 많으면 **대기열에서 기다리게** 하고, 순서대로 출발시킴 (Buffer)

### 왜 필요한가? (3줄 요약)
1. **네트워크 혼잡 방지**: 과도한 트래픽으로 인한 네트워크 마비 예방
2. **QoS 보장**: 중요한 서비스(VoIP, 영상)에 우선순위 부여
3. **SLA 준수**: 고객과 약속한 대역폭 서비스 수준 유지

## 1. 트래픽 폴리싱과 트래픽 쉐이핑의 개념

{: .highlight }
> **트래픽 폴리싱 vs 트래픽 쉐이핑 개념**: Drop vs Buffer → 트래픽 제어 방식 비교
> - (폴리싱) `Drop` (초과 트래픽 즉시 버림)
> - (쉐이핑) `Buffer` (초과 트래픽 저장 후 전송)

### 가. 트래픽 폴리싱(Traffic Policing) 개념

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 네트워크 서비스 보장을 위해 미리 정의된 기준을 초과하는 트래픽은 **버림(Drop)**으로써 트래픽 대역폭을 제어하는 **트래픽 감시기법** |
| **동작방식** | 트래픽이 정해진 대역폭 이상 발생하면 **Drop 처리**하여 대역폭 제어 |
| **적용** | 주로 **Inbound**에 적용, Inbound/Outbound 모두 적용 가능 |

```

│       │                                   │                      │

│       │ ╲  ╱  ╲  ╱  ╲   Traffic          │ ╱         Limited    │
│       │  ╲╱    ╲╱    ╲  Rate             │╱          Traffic    │
│       │                                   │                      │

│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 특징 | 설명 |
|:-----|:-----|
| DoS 공격 차단 | 과도한 트래픽을 즉시 차단하여 DoS 공격 방어 |
| NW 서비스 세분화 | 트래픽 속도 제한 집합을 정의, 수준관리 |
| 엣지 라우터(ER)의 필수 구성 요소 | 네트워크 경계에서 트래픽 제어 |

### 나. 트래픽 쉐이핑(Traffic Shaping) 개념

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 네트워크 성능 최적화를 위해 정해진 트래픽의 초과량을 **버퍼에 저장 및 전송**하는 **트래픽 스무딩 기법** |
| **동작방식** | 트래픽이 대역폭 이상 발생시 **버퍼에 패킷을 저장**후 대역폭 이하 발생시 전송 |
| **적용** | 주로 **Outbound**에 적용, 유입된 Packet을 잠시 저장함 |

```
┌─────────────────────────────────────────────────────────────────┐
│                  트래픽 쉐이핑 개념도                             │

│  Bandwidth(bps)                      Bandwidth(bps)              │
│       │                                   │         Buffered    │
│       │   Offered                         │    ╱╲   ↓          │
│       │   Traffic                         │   ╱  ╲──────────   │
│       │  ╱╲  ╱╲  ╱╲                      │  ╱     Rate         │
│  ─────┼──────────────── Target    ───▶   │ ╱      Shaped       │
│       │ ╲  ╱  ╲  ╱  ╲   Traffic          │╱       Traffic      │
│       │  ╲╱    ╲╱    ╲  Rate             │                      │
│       │                                   │                      │
│       └───────────────▶ time              └───────────────▶ time │
│                                                                  │
│                      Shaping                                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 특징 | 설명 |
|:-----|:-----|
| 전송 속도 정속화 | 일정한 속도로 트래픽 전송 |
| 급격한 변화 방지 | 버스트 트래픽을 부드럽게 처리 |
| 전송 타이밍 관리 | 지연 단축, 패킷손실 저감 |

> 통신 네트워크 서비스 품질과 성능을 일정수준으로 보장하여 대역폭 제한하는 기법

## 2. 트래픽 폴리싱과 트래픽 쉐이핑의 구성요소

{: .highlight }
> **트래픽 폴리싱 vs 트래픽 쉐이핑 구성요소**: Meter/Marker/Drop vs CBS/CIR/EBS/EIR → 구성요소 비교
> - (폴리싱) `MMD` (Meter, Marker, Drop)
> - (쉐이핑) `CCEE` (CBS, CIR, EBS, EIR)

### 가. 트래픽 폴리싱(Traffic Policing)의 구성요소 `MMD`

```
┌─────────────────────────────────────────────────────────────────┐
│                  트래픽 폴리싱 구성요소                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌─────────┐      ┌─────────┐      ┌─────────┐                │
│   │ Packet  │─────▶│  Meter  │─────▶│ Packet  │                │
│   │ Stream  │      └────┬────┘      │  Rate   │                │
│   └─────────┘           │           └────┬────┘                │
│                         │                │                      │
│                         ▼                ▼                      │
│                    ┌─────────┐      ┌─────────┐    ┌────────┐  │
│                    │Overflow │      │Forwarding│───▶│  Mark  │  │
│                    └────┬────┘      └─────────┘    └────────┘  │
│                         │                                       │
│                         ▼                                       │
│                    ┌─────────┐                                  │
│                    │  Drop   │                                  │
│                    └─────────┘                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 구분 | 구성요소 | 설명 |
|:-----|:---------|:-----|
| **기능구성** | **M**eter | 수신되는 패킷의 **유입률을 측정** |
| | **M**arker | 유입률이 약속된 대역폭 이하이면 **마킹 후 포워딩** |
| | **D**rop | 유입률이 약속된 대역폭 이상이면 **패킷을 버림** |
| **작동방식** | 미터링 및 레벨링 | 사전 정의된 대역폭과 비교하고, 초과된 패킷은 **드롭 및 리마킹**하여 우선순위별로 전송 |

> 주로 ISP의 고객회선 용량제한, 네트워크 과부하 방지용으로 사용됨

### 나. 트래픽 쉐이핑(Traffic Shaping)의 구성요소 `CCEE`

```
┌─────────────────────────────────────────────────────────────────┐
│                  트래픽 쉐이핑 구성요소                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│         Packet                                                   │
│         Stream                                                   │
│           │                                                      │
│           ▼                                                      │
│      ┌─────────┐        Overflow        ┌─────────────────┐    │
│      │         │──────────────────────▶│  Excess Overflow │    │
│      │   CBS   │                        │    Discarded     │    │
│      │         │                        └─────────────────┘    │
│      └────┬────┘                                                │
│           │                                                      │
│           ▼ CIR                                                  │
│      ┌─────────┐                                                │
│      │   EBS   │                                                │
│      └────┬────┘                                                │
│           │                                                      │
│           ▼ EIR                                                  │
│      ┌─────────────────┐    ┌─────────────────┐                │
│      │  SLA Guaranteed │    │  Best Effort    │                │
│      │                 │    │     Only        │                │
│      └─────────────────┘    └─────────────────┘                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 구분 | 구성요소 | 설명 |
|:-----|:---------|:-----|
| **기능구성** | **C**BS (Committed Burst Size) | 네트워크에서 전송할 수 있는 **최대 비트 수** (허용 버스트 크기) |
| | **C**IR (Committed Information Rate) | 네트워크에서 항상 보장되는 **평균 전송 속도** (예: 100Mbps, 허용 정보율) |
| | **E**BS (Excess Burst Size) | CBS를 초과한 오버플로우를 버퍼링하는 **추가적 공간크기** (초과 버스트 크기) |
| | **E**IR (Excess Information Rate) | CIR을 넘어서 확장하여 사용할 수 있는 **추가적인 속도** (예: 최대 150Mbps) |
| **작동방식** | 버퍼링 및 포워딩 | 정의된 버퍼용량 외의 패킷은 **저장하고, 추후 전송** |

> ISP, 기업 네트워크, 클라우드 환경에서 대역폭 최적화 및 품질 향상용도로 사용됨

## 3. 트래픽 폴리싱과 트래픽 쉐이핑의 구현 알고리즘

{: .highlight }
> **트래픽 폴리싱 vs 트래픽 쉐이핑 알고리즘**: Drop vs Bucket → 구현 알고리즘 비교
> - (폴리싱) `RWT` (RED, WRED, Tail Drop) + `LTH` (Leaky, Token, Hybrid)
> - (쉐이핑) `LTH` (Leaky Bucket, Token Bucket, Hybrid Bucket)

### 가. 트래픽 폴리싱(Traffic Policing) 알고리즘

| 구분 | 알고리즘 | 동작설명 |
|:-----|:---------|:---------|
| 사전 패킷 폐기 | **R**ED (Random Early Detection) | 트래픽을 구간으로 나누어 최소 임계값까지는 트래픽 드롭이 없지만 **최소 임계값부터 최대 임계값까지 확률적으로 드롭**을 하는 알고리즘 |
| | **W**RED (Weighted Random Early Detection) | RED의 개선한 알고리즘으로, **패킷에 가중치(우선순위)를 부여**하여 Drop 확률을 트래픽 별로 다르게 적용한 알고리즘 |
| | **T**ail Drop | **큐가 가득 차면** 들어오는 트래픽을 무조건 드롭 |
| 버킷 기반 | **L**eaky Bucket | **고정된 속도**로 데이터 전송. 버스트 트래픽이 발생하더라도 초과된 데이터는 버퍼에 저장하거나 폐기 |
| | **T**oken Bucket | **토큰이라는 가상 단위**를 할당 받아 트래픽 전송. 토큰 소진 시 일시적으로 전송 제한. 버스트 트래픽을 허용하면서 평균 트래픽을 제어 |
| | **H**ybrid Bucket | Leaky Bucket와 Token Bucket의 **장점을 결합**한 알고리즘으로 순간적인 버스트 트래픽을 허용하면서 트래픽을 일정한 속도로 방출하는 알고리즘 |

### 나. 트래픽 쉐이핑(Traffic Shaping) 알고리즘 `LTH`

| 구분 | 알고리즘 | 동작설명 |
|:-----|:---------|:---------|
| 용량 제어 | **L**eaky Bucket | **고정된 속도**로 데이터 전송. 버스트 트래픽이 발생하더라도 초과된 데이터는 버퍼에 저장하거나 폐기 |
| 속도 제어 | **T**oken Bucket | **토큰이라는 가상 단위**를 할당 받아 트래픽 전송. 토큰 소진 시 일시적으로 전송 제한. 버스트 트래픽을 허용하면서 평균 트래픽을 제어 |
| 용량+속도 제어 | **H**ybrid Bucket | Leaky Bucket와 Token Bucket의 **장점을 결합**한 알고리즘으로 순간적인 버스트 트래픽을 허용하면서 트래픽을 일정한 속도로 방출하는 알고리즘 |

## 4. 트래픽 폴리싱과 트래픽 쉐이핑 비교

{: .highlight }
> **트래픽 폴리싱 vs 트래픽 쉐이핑 비교**: 버퍼링, 적용, 프로토콜, 알고리즘, 장단점 → 상세 비교
> - (비교) `버버적프알장단`

### 가. 트래픽 폴리싱과 트래픽 쉐이핑 개념 비교

| 구분 | 트래픽 폴리싱 | 트래픽 쉐이핑 |
|:-----|:------------|:------------|
| **개념** | 설정된 대역폭을 초과하는 트래픽을 감지하고 **즉시 폐기(Drop)하거나 리마킹(Remarking)**하여 우선순위를 낮춤 | 설정된 대역폭을 초과하는 트래픽을 **버퍼(Queue)에 저장**했다가 일정한 속도로 전송 |
| **목적** | 네트워크 대역폭 초과 방지, 보안 및 정책 준수 (QoS 정책 적용) | 대역폭 최적화 및 부드러운 트래픽 흐름 유지, 실시간 서비스(VoIP, 스트리밍) 품질 보장 |
| **원리** | **Dropping** | **Buffering** |

### 나. 트래픽 폴리싱과 트래픽 쉐이핑 상세 비교 `버버적프알장단`

| 구분 | 트래픽 폴리싱 | 트래픽 쉐이핑 |
|:-----|:------------|:------------|
| **버**스트 트래픽 | 허용하지 않음 | 일정범위내 허용 |
| **버**퍼링 | 버퍼링 없음, 초과 트래픽 **즉시 폐기** | 버퍼링 존재, 초과 트래픽을 **버퍼에 저장 후 처리** |
| **적**용 | Inbound / Outbound 모두 적용 가능, **주로 Inbound**에 적용 | 유입된 Packet을 잠시 저장함, **주로 Outbound**에 적용 |
| **프**로토콜 | **UDP** 프로토콜에 우수 | **TCP** 프로토콜에 우수 |
| **알**고리즘 | RED, WRED, Tail Drop, Leaky Bucket, Token Bucket, Hybrid Bucket | Leaky Bucket, Token Bucket, Hybrid Bucket |
| **장**점 | 구조 간단, 고속처리, 과부하 방지, 리소스 절약, 패킷 Drop을 통한 Queue 지연 방지 | 대역폭 절감 (TCP 재전송 방지), 실시간 서비스 적합, 초과 패킷 버퍼링 패킷 삭제 가능성 낮음 |
| **단**점 | TCP 기반 애플리케이션의 성능저하 가능, 실시간 서비스(VoIP, 스트리밍 등) 품질저하, 트래픽 변동성(Burstiness)이 증대 가능성 | 버퍼링으로 인한 지연 증가가능성, Burst 트래픽에 제한이 걸릴 수 있음, 일반적으로 송신트래픽에서만 적용 가능, 구현이 어려움 |

> 두 기술의 성능을 비교할 때, 네트워크 환경, 애플리케이션 요구사항, 전체적인 네트워크 관리 목표를 종합적으로 고려

---

## 5. QoS 보장 기술

{: .highlight }
> **QoS 보장 기술**: 큐잉, 가중치, 프로토콜 → QoS 보장 기술
> - (큐잉) `FPCFW` (FIFO, PQ, CBQ, FQ, WFQ)
> - (가중치) `RWT` (RED, WRED, Tail Drop)
> - (프로토콜) `IDM` (IntServ, DiffServ, MPLS)

### 가. QoS 관리기술 종합

| 관리기술 | 기법 | 설명 |
|:--------|:-----|:-----|
| **Queue 관리** | FIFO (First In First Out) | 선입 선출 기반 트래픽 순차 처리 |
| | PQ (Priority Queue) | 우선순위기반 패킷 처리 기준 설정 및 큐잉 |
| | CBQ (Class Based Queue) | 패킷 유형 및 계층 분류 기반 큐 관리 및 통제 |
| | FQ (Fair Queuing) | 큐를 라운드로빈 방식으로 처리 |
| | WFQ (Weighted Fair Queuing) | 패킷 및 큐 단위 가중치 부여 기반 트래픽 처리 |
| **가중치** | RED (Random Early Detection) | 혼잡사항 사전 감지 폐기 |
| | WRED (Weighted-RED) | 패킷의 가중치에 따라 폐기 |
| | Tail Drop (Edge Packet Drop) | 버퍼 임계치 초과시 모두 폐기 |
| **프로토콜** | IntServ | RSVP 프로토콜 기반 라우터 예약 및 경로확보, 전송 |
| | DiffServ | DSCP 기반 TOS(Type Of Service) 마킹 및 전송 |
| | MPLS | L3 계층 IP 패킷 Label 이용한 ISP 레벨 고속 전송 |

> 네트워크 대역폭이 100% 사용되거나, Burst 트래픽이 몰리는 경우, QoS 기술보다 전체적인 대역폭 증설 필요

---

## ⭐ 차별점 키워드

> **Policing = Drop, Shaping = Buffer**: 폴리싱은 초과 트래픽을 **즉시 버리고(Drop)**, 쉐이핑은 초과 트래픽을 **버퍼에 저장 후 천천히 전송(Buffer)**하는 것이 핵심 차이! 폴리싱은 **UDP/Inbound**에, 쉐이핑은 **TCP/Outbound**에 적합!

## 📈 상위 토픽 계층도

```
QoS (Quality of Service)
├── 트래픽 제어 기술
│   ├── 트래픽 폴리싱 (Traffic Policing) ← 현재 토픽
│   │   ├── 구성요소: Meter, Marker, Drop
│   │   └── 알고리즘: RED, WRED, Tail Drop, Bucket
│   └── 트래픽 쉐이핑 (Traffic Shaping) ← 현재 토픽
│       ├── 구성요소: CBS, CIR, EBS, EIR
│       └── 알고리즘: Leaky, Token, Hybrid Bucket
├── Queue 관리 기술
│   ├── FIFO, PQ, CBQ, FQ, WFQ
│   └── RED, WRED, Tail Drop
└── QoS 프로토콜
    ├── IntServ (RSVP)
    ├── DiffServ (DSCP)
    └── MPLS (Label Switching)
```
