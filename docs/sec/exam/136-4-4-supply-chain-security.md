### 🔑 핵심 키워드 3개
- **공급망**: 소프트웨어를 만들 때 사용하는 외부 라이브러리, 도구들
- **제로트러스트**: "아무도 믿지 마라" 원칙

### 📖 등장배경

> SolarWinds, Log4j 같은 사고가 터졌어요! 외부에서 가져온 라이브러리나 도구에 악성코드가 숨어있으면 우리 시스템도 뚫리는 거예요. 그래서 **공급망 보안**이 중요해졌고, **제로트러스트**(아무도 믿지 마라) 원칙을 적용하게 됐어요!

### 📝 개념 정의

> 공급망 보안은 **"외부에서 가져오는 소프트웨어, 하드웨어, 서비스로 인한 보안 위협을 관리하는 것"**이에요.
> 
> 비유하자면, 식재료(라이브러리)가 어디서 왔는지 **원산지**를 확인하고, 상한 재료가 없는지 검사하는 것과 같아요!

### 🏗️ 제로트러스트 핵심 원칙

| 원칙 | 설명 |
|:-----|:-----|
| 무신뢰 | 내부든 외부든 기본적으로 믿지 않음 |
| 항상 검증 | 매번 신원과 권한을 확인 |
| 최소 권한 | 필요한 것만 접근 허용 |

### ⭐ 차별점 키워드

> **Log4j 사고**: 전 세계 시스템에 10년간 영향을 미칠 것으로 예측되는 공급망 공격!

## Ⅰ. 사이버 보안의 최전선, 공급망 필요성

### 가. 공급망 보안의 개념

최근 발표된 **SW 공급망 보안 가이드라인 1.0** 기반 내용 지식 확인을 위한 출제

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     공급망 보안 이슈와 제로트러스트 적용                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   [공급망 보안 이슈]          [기대효과]         [제로트러스트 아키텍처]    │
│                                                                         │
│  • 오픈소스 활용 확산    ───  ✓ 내/외부 보안 강화  ───  • 신원 및 접근관리  │
│  • 큰 피해 규모              ✓ 비인가 접근 차단        (IAM)            │
│    (Solarwinds, Log4j)      ✓ 사고 확산 억제       • 마이크로 세분화    │
│  • 2차 피해 발생             ✓ 규제 대응           • 지속적인 모니터링   │
│    (Codecov 인증정보 유출)   ✓ 리스크              • 보안 자동화        │
│  • 지속적 피해 발생            가시화(SBOM 연계)    • 데이터 보호 및 암호화│
│    (Log4j 10년 영향 예측)                          • 무신뢰 원칙        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

| 구분 | 설명 |
|:-----|:-----|
| **정의** | 조직의 소프트웨어, 하드웨어, 인프라, 서비스 공급자 등 **외부 파트너를 통한 보안 위협**으로부터 정보 자산을 보호하기 위한 **정책, 프로세스, 기술의 통합적 접근 방식** |
| **보호대상** | 제3자 공급자, 소프트웨어 코드 및 라이브러리, CI/CD 파이프라인, 물리적 제품 및 IoT 디바이스 |
| **주요위협** | 악성코드 유입, 코드 변조, 인증정보 유출, 하드웨어 백도어 |

### 나. 공급망 보안의 필요성

**기술적 측면** `외탐위`

| 필요성 | 설명 |
|:------|:-----|
| **외**부 의존성 통제 | 오픈소스, 서드파티 모듈 등 외부 자산에 대한 보안 통제가 필요 |
| **탐**지 및 대응 자동화 | 보안 위협에 대한 실시간 탐지 및 대응 체계가 요구됨 |
| **위**협 경로 차단 | 공급자 경로를 악용한 공격 시나리오 사전 차단 필요 |

**관리적 측면** `책사공`

| 필요성 | 설명 |
|:------|:-----|
| **책**임 범위 명확화 | 사고 발생 시 공급자와 수요자의 책임 범위를 사전에 정의 |
| **사**후 감사 및 검토 체계 | 공급망 관련 보안 로그 및 활동을 정기적으로 검토해야 함 |
| **공**급자 신뢰성 평가 | 공급자의 보안 수준 및 인증 상태를 사전에 확인해야 함 |

### 다. 공급망 보안 관련 주요 표준 및 프레임워크

| 구분 | 세부요소 | 설명 |
|:-----|:-------|:-----|
| **표준** | ISO/IEC 27036 | 공급망 정보보안 관리 표준 |
| **프레임워크** | NIST SP 800-207 | 제로트러스트 아키텍처 공식 가이드 |
| | NIST SP 800-161 | 공급망 보안 위험관리 프레임워크 |
| | MITRE ATT&CK | 공급망 공격 TTP 분석 및 탐지 활용 |
| | SBOM | 소프트웨어 구성 요소 투명성 확보 |
| | CMMC | 미국 국방부 공급망 사이버보안 인증 모델 |

> 공급망 공격은 기업의 통제 밖 자산과 프로세스를 이용한 우회 공격이므로 기존 경계 보안 체계로는 **방어의 한계가 존재**

## Ⅱ. SW 공급망 보안 위험 관리 방안

### 가. 공급망 사이버보안 위험 관리 체계(C-SCRM) 구축 방안

| 구분 | 세부 | 설명 |
|:-----|:-----|:-----|
| **C-SCRM의 정의** | | Cybersecurity Supply Chain Risk Management, 공급망 전체에서 사이버보안 위험을 관리하고 적절한 대응 정책 및 전략 등을 개발하기 위한 체계적 프로세스 |
| **C-SCRM 구축 시 필요 조건** | SDLC 내 통합 | C-SCRM은 설계, 개발, 운영 및 폐기 등 전체 'SW 개발 생명주기(SDLC)'에 걸친 활동을 포괄 |
| | 이해관계자 참여 | 정보보안 및 개인정보보호, 시스템 개발자, 인수, 조달, 법무, 인사(HR) 등 기업 내 다양한 이해관계자 그룹 참여 필수 |
| | 전사 차원 접근 | 공급망 내에서는 적대적, 비적대적 이유로 사이버 위협 발생 가능, C-SCRM을 전사적 위험관리 체계 통합 필요 |

**다단계 전사적 위험 관리 레벨 별 수행 활동**

| 레벨 | 설명 |
|:-----|:-----|
| **레벨-1 (전사)** | 전반적인 C-SCRM 전략, 정책, 구현 계획을 통해 전사적인 C-SCRM으로 관리되는 데 필요한 기조, 거버넌스 구조, 경계를 설정 |
| **레벨-2 (프로세스)** | 레벨-1에서 결정한 전사적인 상황과 방향을 가정하고, 이를 특정한 미션(Mission) 및 비즈니스의 프로세스에 맞게 조정 |
| **레벨-3 (운영)** | 정보시스템이 비즈니스/기능/기술 요구사항을 충족하고, 적절하게 조정된 통제를 포함하는지를 결정하기 위한 기반 제공 |

### 나. SBOM을 활용한 SW 구성 요소의 신뢰성 확보 방안

| 구분 | 세부 | 설명 |
|:-----|:-----|:-----|
| **SBOM의 정의** | | 소프트웨어 컴포넌트 및 구성 요소를 식별할 수 있는 메타데이터와 저작권 및 라이선스 등으로 **소프트웨어 콘텐츠에 대한 정보를 포함하는 공식 SW 자재 명세서** |
| **SBOM 필요성** | SW 구성요소 가시성 확보 | SW에 어떤 구성요소가 존재하는지 신속하게 파악하고, 위험에 대처하기 위한 도구 필요 |
| | 공급망 복잡화 | SW 제작 시 외부 라이브러리나 공개 SW 활용 증가, 다양한 컴포넌트 활용에 따른 SW 공급망 복잡화, 컴포넌트 별 추적 및 관리 필요성 대두 |
| **SBOM 최소 요건** | 데이터 필드 | '공급자명', '타임스탬프', '저작권자', '구성요소명', '버전', '고유식별자', '종속성 관계' 총 7개의 기본 항목 포함 |
| | 자동화 지원 | SPDX, CycloneDX, SWID 등 3가지 포맷을 활용하는 것을 SBOM 공유 및 교환을 위한 자동화 요구 사항으로 정의 |

**SBOM 활용**

| 활용 | 설명 |
|:-----|:-----|
| 공통 취약점 확인 | SBOM 생성 후 SW 구성요소를 식별하고, 'Vulnerabilities' 항목 등에서 알려진 보안취약점 정보(CVE, KEV 등)를 확인 |
| 상용/공개 SW 취약점 확인 | 상용/공개 SW에서 발견된 보안취약점에 대하여 미국 NVD(National Vulnerability Database) 제공 보안취약점 정보 확인 |
| 우선 순위 지정 | 보안취약점 영향 범위와 심각도 등을 활용, 위험 수준 평가 및 조치 우선 순위 지정 |
| 위험도 표현 | 보안취약점 데이터베이스에서 정한 심각도와 기본 점수의 등급으로 위험도 산출 |
| 보안 취약점 개선 | 탐지된 보안취약점에 대한 조치계획을 수립하고, 우선순위에 따라 개발기업, 운영기업(기관) 등에 해당 컴포넌트 정보를 공유 |

> SBOM의 활용을 통해 투명성, 책임성, 식별성, 보안 관리를 위해 개발자, 구매자, 운영자 등 모든 이해관계자에게 긍정적인 효과가 있을 것으로 예상

### 나. 제로트러스트 기반 공급망 보안 아키텍처 구성요소

| 구분 | 구성요소 | 설명 |
|:-----|:--------|:-----|
| **핵심 구성요소** | 정책엔진 (PE) | 접근주체의 리소스 접근 허용 여부 최종결정, 신뢰도 평가 알고리즘 활용 |
| | 정책관리자 (PA) | 주체와 리소스 간 통신 경로 설정, 종료 관리, 세션 별 인증/인가 토큰 또는 크리덴셜 생성 |
| | 정책시행지점 (PEP) | 주체에 할당된 정책 실행, 연결 활성화, 모니터링, 종료, 정책 관리자와 통신, 정책 업데이트 |
| | 접근주체 | 외부벤더, 협력사, 내/외부 사용자 |
| | 보호대상 | 배포서버, Resource서버, 공급망 Repository, Git, Maven |

### 다. 제로트러스트 기반 아키텍처 기술요소 `기정마데모ID`

| 구분 | 기술요소 | 설명 |
|:-----|:--------|:-----|
| 사용자 및 디바이스 | **기**기 신뢰 평가 (Device Posture) | 협력사 단말의 보안 상태 확인 (EDR, NAC) |
| 접근 관리 | **정**책기반 접근 제어 (ABAC/PBAC) | 행위, 역할, 시간, 위치 기반의 세분화된 접근 권한 부여 |
| 네트워크 | **마**이크로세그멘테이션 | 업무 시스템 별 네트워크 분리 및 최소 권한 통신 허용 |
| 데이터 보호 | **데**이터 보호 | 민감 데이터 암호화, DLP, DRM 적용 |
| 모니터링 | **모**니터링 및 대응 | 로그 수집 및 SIEM→이상행위 분석(UEBA)→자동대응(SOAR) |
| 인증 | **ID** 및 접근관리 (IAM) | 제3자 사용자/개체에 대한 인증 강화 (MFA, 연합 인증 등) |

> 제로트러스트 아키텍처는 **지속적 검증, 최소 권한 접근, 세분화된 접근 제어** 등의 3대 원칙을 기반으로 동작

## Ⅳ. 기대효과 및 결론

### 가. 제로트러스트 기반 공급망 보안 기대효과

|:-----|:-----|

| **리스크 가시화** | SBOM 및 접근 로그를 통한 상시 리스크 점검 가능 |
