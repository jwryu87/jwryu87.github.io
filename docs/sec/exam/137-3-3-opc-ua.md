### 한 줄 정의
> OPC Foundation에서 개발한 **플랫폼 독립적이고 안전한 산업 장비 간 데이터 교환**을 위한 국제 통신 표준

### 쉬운 비유 🏭
> **공장의 만국공통어**와 같아요!
> - 🏭 여러 나라(OS) 기계들이 모여있는 공장에서
> - 🗣️ 다 같이 소통할 수 있는 공통 언어(OPC UA)를 만들었어요
> - 🔐 보안 기능까지 내장되어 안전하게 대화해요

### 왜 필요한가? (3줄 요약)
1. **플랫폼 종속성 해결**: 기존 OPC는 Windows 전용이었지만, OPC UA는 모든 OS에서 동작
2. **보안 강화**: 기존 OPC는 암호화 미흡, OPC UA는 X.509 인증서 기반 보안 내장
3. **상호운용성 확보**: 서로 다른 제조사 장비 간 원활한 데이터 교환

## 🎯 기술사 수준 설명

{: .highlight }
> **OPC UA**: 플랫폼 독립성, 상호운용성, 보안 강화 → 산업 자동화 통신 표준
> - (1) `윈암구` → `플보상` (등장배경)
> - (2) `운통H정방보플상` (OPC와 비교)
> - (3) `공산에생품원스클` (활용분야)

## 1. OPC UA 등장배경 `윈암구` → `플보상`

{: .highlight }
> **OPC UA 등장배경**: 플랫폼 독립성, 보안성, 상호운용성 → 산업 자동화 통신 표준
> - (OPC 문제점) `윈암구`
> - (OPC UA 해결) `플보상`
> - (국제표준) `IEC 62541`

### 가. OPC UA의 정의 및 국제 표준

| 구분 | 설명 |
|:-----|:-----|
| **정의** | OPC Foundation에서 개발, 제조현장의 모든 산업 장비와 시스템 간 안전하고 상호 운용 가능한 데이터 교환을 위한 **플랫폼 독립적 국제 통신 표준** |
| **국제 표준** | **IEC 62541** |

### 나. OPC UA 등장배경 `윈암구` → `플보상`

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│       OPC       │     │   OPC 문제점     │     │     OPC UA      │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│ MS 기반         │     │                 │     │                 │
│ COM/DCOM        │ ──▶ │ 윈도우 종속      │ ──▶ │ 플랫폼 독립성    │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│ 산업 시스템     │     │                 │     │                 │
│ 연동            │ ──▶ │ 암호화 미흡      │ ──▶ │ 보안성 개선      │
├─────────────────┤     ├─────────────────┤     ├─────────────────┤
│ 통신 프로토콜   │     │                 │     │                 │
│                 │ ──▶ │ 구성 복잡       │ ──▶ │ 상호 운용성      │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        OPC                  윈암구                   플보상
```

| 구분 | 등장 배경 | 설명 |
|:-----|:----------|:-----|
| **산업 변화 측면** | 기존 OPC Classic의 플랫폼 제약 | 윈도우 전용이었던 OPC Classic과 달리, 다양한 OS 환경에서 시스템 운영 및 연동 필요성 증대 |
| | IoT/스마트공장 확산 | 제조 현장의 자동화 수준 향상, 수많은 센서와 장치에서 발생하는 데이터를 표준화된 방식으로 수집, 통합 및 활용 요구 급증 |
| | 상호운용성 요구 | 서로 다른 제조사의 장비 간 원활한 데이터 교환 및 통신할 수 있는 범용적인 표준의 필요성 대두 |
| **기술 발전 측면** | 보안 기술 강화 | 산업 제어 시스템이 네트워크에 연결되면서 사이버 위협에 대응 위한 강력한 보안 기능을 내재한 기술이 필수 |
| | 복잡한 데이터 모델 지원 | 단순한 '태그' 방식의 데이터 교환을 넘어, 장비의 복잡한 상태, 구성, 이력을 표현 가능한 의미 기반의 정보 모델링 기술이 요구 |
| | 다양한 통신 프로토콜 지원 | TCP/IP를 넘어 HTTP, 웹소켓, MQTT 등 다양한 IT 프로토콜과의 호환 및 클라우드 환경과의 통합 지원하는 기술적 유연성 필요 |

**핵심 요약**: 기존 OPC의 기능이나 성능을 유지하면서 플랫폼 독립성과 상호 운용성 확보를 위하여 OPC UA 등장

## 2. OPC와 OPC UA 비교 `운통H정방보플상`

{: .highlight }
> **OPC vs OPC UA**: Windows vs 멀티플랫폼, COM/DCOM vs SOAP/TCP/IP
> - (비교항목) `운통H정방보플상`

### 가. OPC와 OPC UA 개념 비교

| 구분 | OPC | OPC UA |
|:-----|:----|:-------|
| **개념** | 윈도우 운영체제의 **COM/DCOM 기술**에 기반한 통신 표준 | OPC의 단점을 보완하기 위해 개발, 다양한 운영체제에서 작동하는 새로운 표준 |
| **특징** | 특정 플랫폼에 종속적이며 보안 기능이 미흡 | **플랫폼 독립적**, 강력한 내장 보안 기능을 제공 |

### 나. OPC와 OPC UA 상세 비교 `운통H정방보플상`

| 구분 | OPC | OPC UA |
|:-----|:----|:-------|
| **운**영 체제 | Windows 운영 체제에 국한 | Android, Linux, Windows 포함 **모든 운영 체제** |
| **통**신 | Windows COM, DCOM 기반 | **HTTP/SOAP, TCP/IP** 기반 |
| **H**W 플랫폼 | 컴퓨터 시스템 | PC, Cloud 서버, **PLC, DCS** |
| **정**보 모델 | 각 유형의 데이터 소스를 처리하는 서로 다른 기술 및 모두가 서로 통신할 수 있는 것은 아님 | **하나의 기술로 결합** |
| **방**화벽 | 구성이 복잡하고 취약점 존재 | **설정 간단** |
| **보**안 | 암호화 미흡 | **OpenSSL 인증서**를 통해 접근제어 |
| **플**랫폼 독립성 | Windows 플랫폼 종속 | **플랫폼 독립** |
| **상**호 운용성 | 부족 | **확보** |

### 다. OPC와 OPC UA 세부 사항 비교

| 구분 | OPC | OPC UA |
|:-----|:----|:-------|
| **플랫폼 지원** | **Windows 전용**: COM/DCOM 기반, 제한적 환경: Microsoft 종속성 | **멀티플랫폼**: Windows, Linux, 임베디드, **크로스플랫폼**: OS 독립적 구현 |
| **보안 기능** | **취약한 보안**: 기본 보안 기능 부족, DCOM 보안: 복잡한 설정, 제한적 기능 | **강화된 보안**: X.509 인증서 기반, **다층 보안**: 암호화, 서명, 사용자 인증 |
| **통신 방식** | **클라이언트-서버**: 단일 패턴, 동기식 통신: 실시간 응답 제한 | **다중 패턴**: Client-Server + Pub/Sub, **비동기 통신**: 실시간 데이터 스트리밍 |
| **데이터 모델** | **단순 태그**: 평면적 데이터 구조, 제한된 타입: 기본 데이터 타입만 지원 | **객체지향 모델**: 계층적 구조, **25개 내장 타입**: 복합 데이터 표현 |
| **네트워크 호환성** | **방화벽 문제**: DCOM 포트 관리 복잡, 제한된 프로토콜: TCP/IP 위주 | **방화벽 친화적**: 표준 포트 사용, **다중 프로토콜**: HTTP, WebSocket, MQTT |
| **확장성** | **제한적 확장**: 고정된 사양, 벤더 종속: 개별 구현 방식 | **유연한 확장**: 모듈식 아키텍처, **표준화**: IEC 62541 국제 표준 |

**핵심 요약**: OPC UA에서는 기존 OPC의 단점인 플랫폼 독립성 및 상호 운용성이 개선

## 3. OPC UA 활용분야 `공산에생품원스클`

{: .highlight }
> **OPC UA 활용분야**: 스마트 팩토리, 산업 IoT, 클라우드 연동
> - (활용분야) `공산에생품원스클`
> - (핵심역할) `Industry 4.0`, `스마트 팩토리`

### 가. OPC UA 활용 분야 개요도


### 나. OPC UA 활용 분야 상세 설명 `공산에생품원스클`

| 활용분야 | 특징 | 설명 |
|:---------|:-----|:-----|
| **공**장 자동화 | PLC, 센서, 로봇 | 공장 내 다양한 설비 및 장비 간 데이터 통신 |
| **산**업 IoT | IoT 장치간 연동 | 각종 IoT 디바이스와 게이트웨이의 연결 및 데이터 집계 |
| **에**너지 관리 | 발전, 송/배전 | 스마트 그리드, 발전소, 배전 시스템 등에서 실시간 데이터 교환 |
| **생**산관리 시스템 | 생산 데이터 | MES와 각종 실시간 생산장비 데이터 연동 |
| **품**질 관리 | 품질 모니터링 | 생산기계의 품질 검사 및 모니터링 데이터 집계 |
| **원**격 모니터링/제어 | 원격 환경 | 장비·설비의 상태를 외부에서 모니터링 및 제어 |
| **스**마트 팩토리 | 생산 빅데이터 분석 | 전체 생산라인 통합 관리 및 빅데이터 수집/분석 |
| **클**라우드 연동 | 빅데이터 저장 | 클라우드 기반의 데이터 저장, 분석, 모니터링 시스템과 연결 |

### 다. OPC UA 산업별 활용 분야

| 활용 분야 | 주요 역할 및 특징 | 주요 활용 사례 |
|:---------|:------------------|:---------------|
| **제조업** | Industry 4.0 핵심 기술, 실시간 데이터 수집 | 스마트 팩토리, 예측 유지보수, IoT 장치 통합 |
| **공정 제어** | 안전하고 보안적인 제조 환경 구현 | 석유/가스, 식품/음료, 제약, 자동차 부품 |
| **에너지** | 에너지 시스템 표준화된 데이터 교환 | 전력 시스템, 재생 에너지 통합 |
| **빌딩 자동화** | 지능형 에너지 관리 및 장비 제어 | 조명/공조/보안 시스템, 설비 모니터링 |
| **기계 제조** | 60개 이상 산업장비 표준 데이터 모델 | CNC/공작기계, 사출성형기, 로봇 시스템 |
| **클라우드/IT** | 디지털 트랜스포메이션 인프라 | 클라우드 연동, IIoT, ERP 통합 |

**핵심 요약**: OPC UA는 제조업부터 클라우드 통합까지 다양한 영역에서 활용되고 있으며, 특히 **Industry 4.0**과 **스마트 팩토리** 구현의 핵심 역할을 수행

## ⭐ 차별점 키워드

### 🏆 IEC 62541 (OPC UA 국제 표준)
> OPC UA의 **국제 표준 규격**
> - IEC(국제전기기술위원회)에서 제정한 산업 통신 표준
> - 플랫폼 독립적, 보안 내장, 상호운용성 보장
> - Industry 4.0의 핵심 통신 프로토콜로 채택

### 🏆 Pub/Sub (Publish/Subscribe)
> OPC UA 1.04 버전부터 지원되는 **비동기 통신 패턴**
> - 기존 Client-Server 방식 외에 Pub/Sub 방식 추가
> - 대규모 센서 데이터의 효율적 전송
> - TSN(Time-Sensitive Networking)과 결합 가능
