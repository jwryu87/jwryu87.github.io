### 🔑 핵심 키워드 3개
- **Never Trust**: 절대 신뢰하지 않는다
- **Always Verify**: 항상 검증한다
- **최소 권한**: 꼭 필요한 것만 허용

### 📖 등장배경

> 예전에는 회사 네트워크 안에 있으면 "우리 편"이라고 믿었어요. 하지만 내부에서도 해킹이 일어나고, 재택근무가 늘면서 **"안에 있다고 무조건 믿으면 안 된다"**는 생각이 생겼어요. 그래서 항상 의심하고 검증하는 제로 트러스트가 등장했어요!

### 📝 개념 정의

> 제로 트러스트는 **"아무도 믿지 말고, 매번 확인하라"**는 보안 원칙이에요.
> 
> 비유하자면, 집에 들어올 때마다 **가족이라도 매번 신분증을 확인**하는 것과 같아요!

### 🏗️ 기술요소 (2그룹)

**그룹 1: 핵심 구성요소**

| 구성요소 | 역할 | 쉬운 설명 |
|:--------|:-----|:---------|
| 정책 엔진(PE) | 접근 허용 결정 | 판사 역할 |
| 정책 관리자(PA) | 통신 경로 설정 | 안내원 역할 |
| 정책 시행점(PEP) | 실제 접근 통제 | 경비원 역할 |

**그룹 2: 7가지 원칙**

| 원칙 | 의미 | 쉬운 설명 |
|:-----|:-----|:---------|
| Per-session | 세션마다 검증 | 매번 확인 |
| Dynamic Policy | 상황별 정책 | 유동적 판단 |
| Monitoring | 지속 감시 | 항상 지켜봄 |

### ⭐ 차별점 키워드

> **NIST SP 800-207**: 제로 트러스트의 국제 표준 가이드!

### 📋 6하원칙 요약

| 구분 | 내용 |
|:-----|:-----|
| **누가** | 모든 사용자, 기기, 서비스 |
| **무엇을** | 기업 리소스 접근 |
| **언제** | 매 세션, 매 요청마다 |
| **어디서** | 내부/외부 네트워크 구분 없이 |
| **왜** | 경계 기반 보안의 한계 극복 |
| **어떻게** | 지속 인증, 최소 권한, 마이크로 세그멘테이션 |

## 🎯 기술사 수준 설명

### 📌 핵심 암기 (Quick Reference)

{: .highlight }
> **제로 트러스트**: Never Trust, Always Verify
> - (플레인) `제데` (제어, 데이터)
> - (정책) `PE-PA-PEP`
> - (기술) `마소제` (마이크로세그, SDP, ZT프록시)

---

### 📝 개념 정의

기존 경계 기반 보안 모델의 한계 극복을 위해,
**네트워크 침해 가정, 최소한의 권한(Least Privilege) 부여, 지속적 인증·검증하는 보안 모델**

| 구분 | 설명 |
|:-----|:-----|
| NIST SP 800-207 | 항상 네트워크가 침해되었다 가정, 데이터·서비스 접근에 최소 권한 부여하는 아이디어와 개념의 모음 |
| MIT Lincoln Lab | 악의적 상대에 의해 지속적으로 침해될 수 있는 시스템의 구성요소, 서비스, 사용자를 다루는 일련의 원칙 |
| 핵심 슬로건 | **Never Trust, Always Verify** (절대 신뢰하지 말고, 항상 검증하라) |

┌─────────────────────────────────────────────────────────────────┐

│  │  │Policy Engine │    │Policy Administrator│              │   │
│  │  │    (PE)      │    │       (PA)        │              │   │
│  │  └──────────────┘    └──────────────────┘               │   │

└─────────────────────────────┬───────────────────────────────────┘

┌─────────────────────────────┼───────────────────────────────────┐

│                             ▼                                   │
│  ┌──────┐    ┌─────────────────────┐    ┌────────────────┐     │
│  │비신뢰 │───▶│ Policy Enforcement  │───▶│  Enterprise    │     │
│  │ 주체  │    │    Point (PEP)      │    │   Resource     │     │

         └─────────────────────────────────┘

### 📊 정책 구성요소 `PE-PA-PEP`

| 구성요소 | 풀네임 | 역할 |
|:--------|:------|:-----|
| **PE** | Policy Engine | 다양한 입력 요소 검토, 자원에 대한 접근 허용 여부 **결정** |
| **PA** | Policy Administrator | 주체와 자원간 통신 경로 **설정 및 종료 관리** |
| **PEP** | Policy Enforcement Point | 주체에 할당된 정책 **실행**, 연결 활성화, 모니터링, 종료 |

**접근 결정 데이터 입력 요소**

| 요소 | 역할 |
|:-----|:-----|
| 규제/내부규정 | 법적 규제·기업 내부 규정 준수 확인 |
| 데이터 접근 정책 | 리소스 접근에 대한 속성, 규칙, 정책 |
| 보안 정보/이벤트(SIEM) | 보안정보 수집, 정책 개선, 공격 경고 |
| 위협 인텔리전스 | 내/외부 발생 보안 위협 정보 |
| ID 관리 시스템 | 사용자 계정·식별 기록 관리 |
| 행위로그 | 각종 로그 및 분석 결과 |

---

### 📜 7가지 원칙 (NIST SP 800-207)

| 원칙 | 내용 |
|:-----|:-----|
| **Resources** | 모든 데이터·컴퓨팅 서비스는 자원으로 간주, SaaS·소형기기 포함 |
| **Communication** | 모든 통신은 환경에 관계 없이 동일한 보안 요구 충족, 내부 NW도 예외 없음 |
| **Per-session Access** | 조직 리소스에 대한 접근은 **개별 세션마다** 허가 필요 |
| **Dynamic Policy** | 클라이언트 식별자, 응용, 자산 상태, 행동, 환경 속성에 따라 **동적 정책** 결정 |
| **Monitoring** | 기본적으로 자산 불신, 리소스 접근 판단 시 자산 상태 **지속 감시** |
| **Authentication & Authorization** | 모든 리소스에 정형화된 사이클로 접근, ICAM, 다중 인증 권장 |
| **Continuous Improvement** | 보안 상태, 트래픽, 접근 요청 정보 수집, **지속적 보안 개선** |

---

### 🔧 핵심 기술 `마소제`

| 기술 | 설명 |
|:-----|:-----|
| **마**이크로 세그멘테이션 | VLAN, VxLAN 통해 서비스 논리적 그룹으로 분류 |
| **소**프트웨어 정의 경계 (SDP) | 애플리케이션-사용자 간 IP터널 설정 (IPSec, TLS VPN), SDDC 활용 |
| **제**로 트러스트 프록시 | 인증된 사용자-앱의 주문형 경계 설정, IAP(Identity Aware Proxy) |

---

### 🏛️ 배치 모델

| 모델 | 설명 |
|:-----|:-----|
| 기기 에이전트-게이트웨이 | 기기 에이전트와 리소스 게이트웨이가 PEP 구성, 강력한 기기 관리 기업에 적합 |
| 리소스 그룹 배치 | 게이트웨이가 리소스 그룹 앞에 위치 |
| 리소스 포탈 배치 | 에이전트 없이 게이트웨이 포탈로 접근 |
| 기기 응용 샌드박스 | 기업 승인 신뢰 응용을 샌드박스에서 실행 |
