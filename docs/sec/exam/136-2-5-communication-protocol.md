## 1. 통신 프로토콜의 개념

{: .highlight }
> **통신 프로토콜 개념**: 구문, 의미, 타이밍
> - (구성요소) `구의타`
> - (제어 프로토콜) `오혼흐`
> - (고려사항) `성신호`

### 가. 통신 프로토콜의 정의

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 서로 다른 장치나 시스템 간 데이터를 송수신하기 위해, **구문, 의미, 타이밍** 등의 규칙과 제어 절차를 정의한 **통신 규약** |
| **목적** | 데이터 전송의 신뢰성 강화 위해 오류/혼잡/흐름 제어 규약 포함 |

### 나. 통신 프로토콜 개념도

│                   │                   │                          │

| **구**문 (Syntax) | 데이터 형식, 코딩, 신호 레벨 등 구조적 형식 정의 |
| **의**미 (Semantics) | 제어 정보, 오류 처리 등 데이터의 의미 정의 |
| **타**이밍 (Timing) | 속도 조절, 순서 제어 등 시간적 순서 정의 |

## 2. 오류 제어, 혼잡 제어

{: .highlight }
> **오류 제어, 혼잡 제어**: 신뢰성 확보, 전송 효율성 확보
> - (오류 제어) `FB` (FEC, BEC)
> - (혼잡 제어) `SCFR` (Slow Start, Congestion Avoidance, Fast Retransmit, Fast Recovery)

### 가. 신뢰성 확보, 오류제어

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 데이터 전송 과정에서 발생하는 비트 오류, 패킷 손실을 **검출·복구**하여 전송 신뢰성을 확보하는 기술 |

```
┌─────────────────────────────────────────────────────────────────┐
│                        오류 제어 개념도                          │
├────────────────────────────┬────────────────────────────────────┤
│           <FEC>            │              <BEC>                  │
├────────────────────────────┼────────────────────────────────────┤
│                            │                                     │
│   송신측        수신측      │    송신측           수신측          │
│     │             │        │      │                │            │
│     │    오류     │        │      │     오류       │            │
│     ├─────▶ ★ ───▶│        │      ├─────▶ ★ ──────▶│            │
│     │             │        │      │                │            │
│     │         오류 정정    │      │◀── 재전송요청 ──│            │
│     │             ▼        │      │                │            │
│                            │      ├── 재전송 ─────▶│            │
│                            │      │            오류 검출        │
└────────────────────────────┴────────────────────────────────────┘
```

#### 오류 제어 유형 `FB`

| 유형 | 기법 | 설명 |
|:-----|:-----|:-----|
| **F**EC (전진 오류 정정) | Hamming Code | 데이터 비트 지정 위치에 패리티 비트 삽입한 오류 정정 코드 |
| | Reed Solomon Code | 다중 비트 오류 검출 정정 코드 |
| | 길쌈코드, 터보코드 | 입력비트와 과거 상태 반영하여 출력 하는 정정 코드 |
| **B**EC (후진 오류 정정) | 체크섬 | 4 계층(TCP) 오류 제어 검출 기법 |
| | 패리티 검사, CRC | 2 계층 오류제어 검출기법 |
| | ARQ | Stop-and-wait, Go-Back-n, Selective Repeat |

> 오류 제어 바탕으로 혼잡 제어 통해 안전성 극대화 필요

### 나. 전송 효율성 확보, 혼잡제어

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 네트워크 자원이 한계에 도달했을 때 패킷 손실과 지연을 최소화하고, **전송 효율성을 유지**하는 기술 |

```
┌─────────────────────────────────────────────────────────────────┐
│                      혼잡 제어 개념도                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Window                    Congestion                            │
│  Size                      Avoidance        Fast                 │
│    │                           │          Recovery               │
│    │                           │              │                  │
│    │    Slow                   │    Fast      │                  │
│    │    Start                  │  Retransmit  │                  │
│    │      ╱                    │     │        │                  │
│ ssthresh1├─────────────────────┼─────┤        │   Slow          │
│ ssthresh2├─────────────────────┴─────┼────────┤   Start         │
│ ssthresh3├───────────────────────────┴────────┼──────           │
│    │                                          │                  │
│    └──────────────────────────────────────────┴─────────▶ Time  │
│                    3 Duplicate  timeout                          │
│                        ACK                                       │
└─────────────────────────────────────────────────────────────────┘
```

#### 혼잡 제어 동작절차 `SCFR`

| 단계 | 동작절차 | 설명 |
|:-----|:---------|:-----|
| 1 | **S**low Start | Congestion Window(CWND)를 **지수적으로 증가**시키는 기법 |
| 2 | **C**ongestion Avoidance | CWND > ssthresh(임계값) 도달시 진입, **선형적 크기 증가** |
| 3 | **F**ast Retransmit | Duplication ACK **3회** 수신시 해당 세그먼트 **빠른 재전송** |
| 4 | Fast **R**ecovery | 해당 CWND 크기에서 바로 **선형 증가**하며 전송 |

#### 혼잡 제어 유형

| 유형 | 알고리즘 | 설명 |
|:-----|:---------|:-----|
| **손실기반 혼잡제어** | Tahoe, Reno, NewReno, BIC, CUBIC, H-tCP | 패킷 손실, 타임 아웃, ACK 손실 발생시 혼잡상황으로 판단하고 혼잡 윈도우 크기 조절 |
| **지연기반 혼잡제어** | TCP Vegas, Westwood, Fast TCP | 측정한 RTT (Round Trip Time) 값이 늘어나는 경우 혼잡 상황으로 판단하고 혼잡 윈도우 크기 조절함 |

> 혼잡 제어 통해 패킷 손실 최소화 및 전송 효율 강화 가능

## 3. 슬라이딩 윈도우(Sliding Window) 기법

{: .highlight }
> **슬라이딩 윈도우**: 데이터 흐름제어, 동적 윈도우
> - (개념) `수신측 윈도우 크기` = min(CWND, RWND)
> - (동작절차) `설전닫열`

### 가. 슬라이딩 윈도우 개념

| 구분 | 설명 |
|:-----|:-----|
| **개념** | 수신 측에서 설정한 **윈도우 크기**만큼 송신 측에서 확인 응답(ACK)없이 전송할 수 있게 하여 **흐름을 동적으로 조절**하는 제어 알고리즘 |
| **윈도우 크기** | min(CWND, RWND) - CWND: Congestion Window, RWND: Receive Window |

```
┌─────────────────────────────────────────────────────────────────┐
│                  슬라이딩 윈도우 개념도                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                     윈도우 크기                                  │
│                  MIN(CWND, RWND)                                │
│            ◀─────────────────────▶                              │
│                                                                  │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐       │
│  │  1  │  2  │  3  │  4  │  5  │  6  │  7  │  8  │  9  │       │
│  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘       │
│    ACK    ACK    즉시 전송 가능      ACK 수신 후                │
│   수신완료  미수신     데이터       전송가능 데이터              │
│           (전송된)                                               │
│      │                         │                                │
│      ▼                         ▼                                │
│    닫힘                       열림                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

> 송신 장치와 수신 장치간 전송 속도 차이 극복 위한 흐름 제어 기법

### 나. 슬라이딩 윈도우 동작 절차 `설전닫열`

| 구분 | 동작 절차 | 설명 |
|:-----|:---------|:-----|
| 윈도우 **설**정 | 1. 윈도우 크기 설정 | min(CWND, RWND) 크기로 윈도우 크기 설정 |
| 데이터 **전**송 | 2. 윈도우 크기 내 전송 | 송신측에서 윈도우 크기 내 자유롭게 데이터 전송 |
| 윈도우 이동 | 3. **닫**힘 동작 | ACK 수신 시 왼쪽 경계를 오른쪽으로 이동 |
| | 4. **열**림 동작 | 줄어든 윈도우를 윈도우 크기만큼 늘림, 오른쪽 경계 오른쪽 이동 |

> 슬라이딩 윈도우 사용 시 **Silly Window Syndrome**과 같은 비효율적 전송 발생하여 송신측 버퍼 활용하여 해결 필요. **네이글(Nagle) 알고리즘**, **클락(Clark) 알고리즘** 이용하여 Silly Window Syndrome 방지 가능

## 4. 프로토콜 설계 시 고려사항

{: .highlight }
> **프로토콜 설계 시 고려사항**: 호환성, 안전성, 성능, 효율성
> - (고려사항) `호안성효`
> - (호환성) `표레확상`
> - (안전성) `오혼데암`
> - (성능) `혼흐데확`
> - (효율성) `오자전적`

### 가. 호환성 및 안전성 측면 고려사항

| 구분 | 고려사항 | 설명 |
|:-----|:---------|:-----|
| **호**환성 측면 | **표**준 준수 | IEEE, 3GPP 등 국제 표준 규격 준수 |
| | **레**거시 프로토콜 지원 | 기존 프로토콜 체계 호환, 충돌 여부 검토 필요 |
| | **확**장성 | 향후 기능 추가 버전 상향 시 호환성 확보 |
| | **상**호운용성 | 타 장비, 타 프로토콜과의 상호 동작 가능성 확보 |
| **안**전성 측면 | **오**류 제어 적용 | 데이터 신뢰성 확보 |
| | **혼**잡 제어 적용 | 전송 신뢰성, 패킷 손실 방지 |
| | **데**이터 무결성 확보 | 디지털 서명, HASH 기법 적용 |
| | **암**호화 지원 | TLS, E2EE 등 암호화 기법 적용 |

> 호환성, 안전성 선확보 후 성능 및 효율성 강화 고려

### 나. 성능 및 효율성 측면 고려사항

| 구분 | 고려사항 | 설명 |
|:-----|:---------|:-----|
| **성**능 측면 | **혼**잡 제어 (Congestion Control) | 네트워크 혼잡 상황에서 전송률을 조절하여 성능 저하 및 패킷 손실을 최소화 |
| | **흐**름 제어 (Flow Control) | 송·수신자 간 속도 차이를 조정하여 수신 측 버퍼 오버플로우 방지, 전송 효율 극대화 |
| | **데**이터 전달 방식 (Data Delivery Method) | 연결형(신뢰성 중심) 또는 비연결형(속도 중심) 전달 방식을 선택해 네트워크 성능 최적화 |
| | **확**장성 및 상호운용성 (Scalability & Interoperability) | 대규모 네트워크 확장에도 안정적인 동작을 보장, 다양한 시스템·네트워크 환경에서 호환성 확보 (예: IPv4↔IPv6) |
| **효**율성 측면 | **오**버헤드 최소화 | 헤더 압축, HPACK, QPACK |
| | **자**원 최적화 | 시간, 대역폭 자원 분할 최적화 |
| | **전**력 효율 달성 | 저전력 장비, 백스케터링 |
| | **적**응성 강화 | AI 통한 예측, 빠른 적응성 |

> AI 내재화 통한 통신 프로토콜 성능, 효율 최적 설계 필요

### 다. 안정성 및 보안 측면 고려사항

| 구분 | 고려 사항 | 설명 |
|:-----|:---------|:-----|
| 안정성 측면 | 데이터 **기**밀성 (Confidentiality) | 전송 구간 암호화를 통해 데이터가 제3자에게 노출되지 않도록 보호 |
| | **무**결성 (Integrity) | 데이터가 위·변조되지 않았음을 보장하기 위해 해시, MAC, 디지털 서명 등을 적용 |
| | **인**증 (Authentication) | 사용자·단말·시스템의 신원을 검증하여 비인가 접근을 차단 |
| 보안 측면 | **가**용성 (Availability) | 서비스 중단 없이 안정적으로 제공하기 위해 QoS 보장, DDoS 방어 등 적용 |
| | **접**근통제 및 권한관리 (Access Control) | 네트워크 자원에 대한 권한을 최소화하고 불법적 접근 차단 |

> 통신 프로토콜은 신뢰성 있는 데이터 전송을 목표로, 오류·혼잡 제어와 같은 핵심 기능, 이를 구현하는 기법, 설계 원칙을 함께 고려해야 함

## ⭐ 차별점 키워드

### 🏆 TCP Tahoe vs TCP Reno
> **손실 기반 혼잡 제어 알고리즘의 진화**
> - **Tahoe (초기버전)**: 패킷 손실 시 CWND를 1로 리셋, Slow Start부터 다시 시작
> - **Reno (이후 버전)**: Fast Recovery 도입, ssthresh를 절반으로 설정하고 선형 증가
> - **CUBIC**: Linux 기본 알고리즘, 3차 함수 기반 윈도우 크기 조절

### 🏆 Silly Window Syndrome
> **슬라이딩 윈도우의 비효율적 전송 문제**
> - 수신 측 버퍼가 작을 때 송신 측이 작은 세그먼트를 반복 전송하는 현상
> - **네이글(Nagle) 알고리즘**: 송신 측에서 작은 패킷을 모아서 전송
> - **클락(Clark) 알고리즘**: 수신 측에서 버퍼가 충분할 때만 윈도우 크기 공지

## 📈 상위 토픽 계층도

```
네트워크
├── OSI 7 Layer
│   ├── 물리 계층
│   ├── 데이터링크 계층
│   │   └── 오류 제어 (FEC, BEC)
│   ├── 네트워크 계층
│   └── 전송 계층
│       ├── TCP/UDP
│       ├── 오류 제어 ← 현재 토픽
│       ├── 혼잡 제어 ← 현재 토픽
│       └── 흐름 제어 (슬라이딩 윈도우) ← 현재 토픽
├── 프로토콜 설계
│   ├── 호환성
│   ├── 안전성
│   ├── 성능
│   └── 효율성
└── TCP 혼잡 제어 알고리즘
    ├── Tahoe / Reno
    ├── CUBIC
    └── BBR
```
