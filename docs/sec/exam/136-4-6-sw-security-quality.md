### 쉬운 비유 🏰
> **성(城)의 방어력 품질**과 같아요!
> - 🔒 **기밀성**: 허가된 사람만 성에 들어갈 수 있음

> - 📜 **부인방지**: 누가 언제 출입했는지 기록
> - 🎖️ **책임성**: 경비원의 책임 추적 가능
> - 🔑 **인증성**: 출입증으로 신분 확인

### 왜 필요한가? (3줄 요약)
1. **사이버 공격 증가**: 제품 출시 전 보안 결함 제거 필수
2. **경영 리스크**: 보안 취약점은 기능보다 심각한 경영 리스크와 피해 유발
3. **규제 요건**: ISMS, ISO/IEC 27001, 공공 발주 요건에서 보안은 기본 품질 요구사항

## 1. 소프트웨어 보안 품질의 정의와 중요성

{: .highlight }
> **보안 품질 정의**: 기밀성, 무결성, 부인방지, 책임성, 인증성
> - (정의) `기무부책인` (ISO 25010 보안성 부특성)
> - (중요성-공급자) `법신효`
> - (중요성-사용자) `안기서`

### 가. 소프트웨어 보안 품질의 정의 `기무부책인`

| 구분 | 정의 |
|:-----|:-----|
| **정의** | 소프트웨어의 보안 특성인 **기밀성, 무결성, 부인방지, 책임성, 인증성**의 속성을 충족시켜 적합한 사용자에게 안전한 데이터 접근과 활용을 보장하는 소프트웨어 제품의 **정보 및 데이터 보호 수준** |

**ISO 25010 기준 보안성 주특성 아래 5가지 부특성:**

| 부특성 | 설명 |
|:-------|:-----|
| **기**밀성 (Confidentiality) | 인가된 사용자만 데이터에 접근 가능하도록 보장 |
| **무**결성 (Integrity) | 데이터가 무단으로 변경되지 않았음을 보장 |
| **부**인방지 (Non-Repudiation) | 행위나 이벤트가 발생했음을 부인할 수 없도록 증명 |
| **책**임성 (Accountability) | 시스템 내 행위에 대한 책임 추적 가능성 보장 |
| **인**증성 (Authenticity) | 사용자 또는 리소스의 신원을 증명 |

### 나. 소프트웨어 보안 품질의 중요성

| 관점 | 중요성 | 설명 |
|:-----|:-------|:-----|
| **소프트웨어 공급자 관점** | 보안 관련 **법**적, 규제 준수 | 소프트웨어에 필요한 개인정보보호 및 보안 관련 의무사항에 대한 사전 준수를 통해 법적 리스크 최소화 |
| | 고객 **신**뢰도 상승, 시장 경쟁력 강화 | 고객의 신뢰 바탕으로 소프트웨어의 경쟁력을 증명하여 시장에서의 경쟁 우위 확보 |
| | 보안 유지보수 **효**율화 | 보안 사고 예방을 통해 취약점 발견 시 발생할 수 있는 추가 보안 개발 및 대응 비용 사전 차단 |
| **소프트웨어 사용자 관점** | **안**전한 소프트웨어 이용 | 보안 취약점을 사전 예방한 제품을 사용함으로써 소프트웨어 제품의 안전한 이용 가능 |
| | 데이터 **기**밀성, 무결성 | 소프트웨어에서 사용하는 개인정보 및 민감정보에 대한 기밀성, 무결성 보장 |
| | **서**비스 연속성 보장 | 소프트웨어에 대한 보안적 리스크를 사전에 해결하여 해당 제품으로 인한 비즈니스 중단요소 제거 |

> 광범위한 부분의 취약점을 점검하고, 빠르게 대응방안을 적용해야 하는 보안 특성 상 자동화 기술과 도구 필수

## 2. 보안 품질 확보를 위한 자동화 기술과 도구

{: .highlight }
> **자동화 기술과 도구**: 정적분석, 동적분석, 구성분석, CI/CD, 코드서명
> - (기술) `정동구CI코`
> - (SAST) `SonarQube`, `Fortify`, `Checkmarx`
> - (DAST) `OWASP ZAP`, `Burp Suite`
> - (SCA) `Snyk`, `Black Duck`

### 가. 자동화 기술과 도구 `정동구CI코`

| 자동화 기술 | 도구 | 설명 |
|:-----------|:-----|:-----|
| **정**적분석 (SAST) | SonarQube | 20개 이상의 프로그래밍 언어를 지원하며, 버그 및 코드 스멜, 보안 취약점 분석 가능 |
| | Fortify | 보안 중심의 정적분석에 특화된 정적분석도구 |
| | Checkmarx | 웹 기반 분석, 클라우드 앱 보안에 적합한 정적분석도구 |
| **동**적분석 (DAST) | OWASP ZAP | OWASP에서 개발한 오픈소스 웹 어플리케이션 보안 취약점 스캐너 |
| | Burp Suite | 약점 자동 분석(스캐닝), 브루트포스 공격 등의 다양한 기능을 제공하는 동적분석도구 |
| **구**성 분석 (SCA) | Snyk | 오픈소스 라이브러리 취약점 탐지, 컨테이너 및 클라우드 인프라 보안에 특화된 SCA 솔루션 |
| | Black Duck | 소프트웨어 구성 분석 및 오픈소스 관리 자동화 |
| **CI**/CD 보안 자동화 | GitHub Actions + SAST/DAST | GitHub에서 공식적으로 제공하는 Workflow 자동화 지원 도구, CI/CD 파이프라인에 보안 자동화 도구 연동 지원 |
| | Jenkins + SAST/DAST | 오픈소스 CI/CD 자동화 도구, 보안 자동화 도구 연동 지원 |
| | Trivy | 컨테이너 이미지 취약점 스캐너 |
| **코**드서명 | Sigstore | 소프트웨어, 산출물에 대한 디지털 서명지원, 배포된 소프트웨어의 무결성 보장 |
| | MS SignTool | Window 소프트웨어의 디지털 서명 지원 도구 |

> 광범위한 부분의 취약점을 점검하고, 이에 대한 대응방안을 적용해야 하는 특성 상 자동화 기술 및 도구 필수

### 나. 라이프사이클 단계별 보안 품질 확보 자동화 구현방안 `요설개테배`

| | 보안 요구사항 관리 자동화 | 변경내역에 대한 자동 관리를 통한 추가 보안요구사항 자동 추출 및 적용 |

| | 시큐어 코딩 템플릿 | 시큐어 코딩 가이드에 명시된 규칙으로 작성된 Class, Method 템플릿 제공 |
| **테**스트 | DAST 자동화 | 보안 테스트 케이스 적용, 취약점 리포팅 자동화. OWASP ZAP 등의 DAST 도구 연동 |
| | 보안 테스트 시나리오 자동생성 | 보안 공격 시나리오에 따른 소프트웨어의 취약요소 검증 자동화. Selenium과 같은 소프트웨어 자동수행 기술 적용 |
| **배**포 및 운영 | 코드서명 자동화 | 배포 전 자동화 도구를 연동하여 배포물의 위변조 방지, 무결성 보장. CD 파이프라인에 Sigstore, MS SignTool 등의 코드서명 도구 연동 |
| | 컨테이너 이미지 스캔 | 배포 시 컨테이너 이미지에 대한 보안 검증. CD 파이프라인에 Trivy 등의 도구 연동 |
| | 취약점, 공격 행위 탐지 자동화 | 지속적인 모니터링을 통한 위협 탐지 및 대응. Splunk, ELK와 같은 로그 분석 기술 적용 |

> 최근 지속적인 IT 시스템의 보안사고에 대한 실질적인 대응 방안으로 자동화된 보안 품질 확보 필요성 대두

## 4. 보안 품질 확보 자동화의 기대효과와 시사점

{: .highlight }
> **기대효과와 시사점**: 비즈니스/기술 관점, 긍정적/부정적 시사점
> - (기대효과) `비기` (비즈니스, 기술)
> - (긍정적 시사점) `DevSecOps`, `생태계`, `전문성`
> - (부정적 시사점) `의존`, `비용`, `충돌`

### 가. 보안 품질 확보 자동화의 기대효과 `비기`

| 관점 | 기대효과 | 설명 |
|:-----|:---------|:-----|
| **비**즈니스 관점 | 보안 비용 절감 | 보안 테스트 시 발생하는 인력, 시간 등의 비용 효율화 |
| | 보안 사고 예방 | 보안 사고로 인한 서비스 중단 요소 사전 방지 |
| | 제품 출시 기간 단축 | 테스트 및 점검에 필요한 시간을 단축, 빠른 제품 출시 |
| **기**술 관점 | 정량적 보안관리 가능 | 자동화된 도구를 통한 보안 상태 시각화 및 수치화 가능 |
| | 보안 품질 표준화 | 개발자 편차 없이 일관된 보안정책 적용 |
| | 지속적 보안통제 | 개발 및 운영, 신규 기능 업데이트 시에도 자동화된 테스트가 가능하여 지속적인 보안 통제 가능 |

> 보안 품질 확보를 위한 자동화 도입은 조직에 긍정적, 부정적 영향이 동시 존재, 이에 대한 분석 및 검토 필요

### 나. 보안 품질 확보 자동화의 시사점

| 관점 | 시사점 | 설명 |
|:-----|:-------|:-----|
| **긍정적 시사점** | DevSecOps 문화 정착 | 개발 전반의 보안 의식강화를 통한 고품질 제품 생산 |
| | 보안 기술 생태계 조성 | 파편화된 보안 기술을 통합하여 자동화할 수 있는 생태계 조성을 통한 효율성 및 기술력 강화 증진 |
| | 보안 인력의 전문성 강화 | 보안 인력이 단순 수명업무가 아닌 분석 및 대응 전략 업무에 집중할 수 있는 환경 조성 |
| **부정적 시사점** | 과도한 자동화 의존 | 자동화 도구가 탐지하지 못하는 영역에 대한 보안 Gray Zone 발생 가능성 존재 |
| | 추가 비용 발생 | 상용 도구 라이센스, 추가 인프라 구축/운영 비용 발생 |
| | 기존 개발 환경과의 충돌 | 기존 개발환경과 충돌하여 생산성 저하를 유발할 수 있으며, 기존 개발자들의 저항 발생 가능 |

> 점진적인 보안 자동화 도입과 필수 보안 프로세스 체계적 확립을 통해 부정적 시사점을 효과적으로 최소화

## ⭐ 차별점 키워드

### 🏆 DevSecOps (Development + Security + Operations)
> **개발-보안-운영 통합 프레임워크**
> - 소프트웨어 라이프사이클 전체에 보안을 내재화
> - **Shift-Left Security**: 보안을 개발 초기 단계로 이동
> - CI/CD 파이프라인에 보안 자동화 도구 연동
> - 지속적인 보안 테스트 및 모니터링

### 🏆 ISO 25010 보안성 (Security)
> **소프트웨어 품질 표준의 보안 특성**
> - 8가지 품질 주특성 중 하나로 보안성 포함
> - 5가지 부특성: 기밀성, 무결성, 부인방지, 책임성, 인증성
> - 소프트웨어 보안 품질의 국제 표준 근거
