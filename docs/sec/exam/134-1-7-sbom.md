### 🔑 핵심 키워드 3개
- **SBOM**: 소프트웨어에 들어있는 부품 목록서
- **메타데이터**: 부품을 식별할 수 있는 정보
- **공급망 보안**: 소프트웨어가 어디서 왔는지 추적하는 것

### 📖 등장배경

> 소프트웨어는 마치 자동차처럼 여러 부품(라이브러리, 프레임워크)을 조립해서 만들어요. 그런데 그 부품 중에 해킹당한 위험한 부품이 들어있으면 어쩌죠? 그래서 **"이 소프트웨어에는 뭐가 들어있어요?"**라고 물어봤을 때 정확히 알려줄 수 있는 **부품 목록서(SBOM)**가 필요해졌어요. 미국에서는 2021년부터 소프트웨어를 팔 때 이 목록서를 제출하도록 의무화했어요!

### 📝 개념 정의

> SBOM은 **"소프트웨어에 들어있는 모든 부품을 기록한 목록서"**예요.
> 
> 비유하자면, **자동차 부품 목록서**처럼 어떤 부품이 어디서 왔는지, 버전은 뭔지, 누가 만든 건지 다 적어놓은 거예요!

### 🏗️ 기술요소 (2그룹)

**그룹 1: SBOM 구성요소**

| 구성요소 | 쉬운 설명 | 예시 |
|:---------|:---------|:-----|
| 작성자 정보 | 누가 만들었는지 | Acme, Bob |
| 업데이트 시간 | 언제 마지막으로 수정했는지 | 2024-01-15T10:30:00Z |
| 공급업체 이름 | 어느 회사에서 만든 건지 | Acme, Bingo |
| 구성요소 이름 | 부품 이름 | Application, Browser |
| 버전 정보 | 몇 버전인지 | V1.1, V2.2 |
| 해시 값 | 부품이 변조되지 않았는지 확인 | 0x123, 0x223 |

**그룹 2: 표준 포맷**

| 포맷 | 만든 곳 | 용도 |
|:-----|:--------|:-----|
| SPDX | 리눅스 재단 | 오픈소스 저작권·라이선스 정보 교환 |
| CycloneDX | OWASP 재단 | 공급망 보안 및 경량 SBOM |
| SWID | - | SW 정보 태그 생성 및 인벤토리 |

### ⭐ 차별점 키워드

> **공급망 보안의 핵심**: 소프트웨어에 뭐가 들어있는지 모르면 보안 취약점을 찾을 수 없어요. SBOM으로 모든 부품을 추적할 수 있어요!

### 📋 6하원칙 요약

| 구분 | 내용 |
|:-----|:-----|
| **누가** | 소프트웨어 개발자, 공급업체 |
| **무엇을** | 소프트웨어 구성요소 목록서(SBOM) |
| **언제** | 2021년 미국에서 제출 의무화, 개발 시점부터 작성 |
| **어디서** | 소프트웨어 개발·배포 과정 |
| **왜** | 공급망 보안, 취약점 관리, 라이선스 컴플라이언스 |
| **어떻게** | 메타데이터 수집 → 표준 포맷(SPDX/CycloneDX/SWID) 작성 → 자동화 도구 활용 |

## 🎯 기술사 수준 설명
{: #-기술사-수준-설명}

{: .highlight }
> **SBOM**: 메타데이터, 저작권라이선스, 공식자재명세서 → SW 구성요소 목록 관리 기법
> - (구성요소) `작시공구버해고관`
> - (표준포맷) `S-C-S`
> - (고려사항) `관기`

---

### 📝 개념 정의

SW 공급망 관리의 중요성에 따른 출제 예상으로, SW 공급망 공격 대응을 위한 기술이다.

소프트웨어 컴포넌트 및 구성 요소를 식별할 수 있는 메타데이터와 저작권 및 라이선스 등으로,
소프트웨어 콘텐츠에 대한 정보를 포함하는 공식 SW 자재 명세서이다.

**등장배경:**
- SW 공급망 공격 증가 (SolarWinds, Codecov 등)
- 2021년 미국 행정명령: SW 제품에 대한 SBOM 제출 의무화
- 글로벌 시장 진출을 위한 SBOM 적용 필요성

---

### 🏗️ SBOM 구성요소 `작시공구버해고관`

| 구분 | 핵심 기술 | 설명 |
|:-----|:----------|:-----|
| **작**성자 정보 (Author Name) | Baseline | SW 작성자 정보 |
| **시**각 (Timestamp) | Baseline | SBOM이 마지막으로 업데이트된 날짜 및 시간 (ISO 8601) |
| **공**급업체 이름 (Supplier Name) | Attributes | SW 공급업체의 이름 또는 기타 식별자 |
| **구**성요소 이름 (Component Name) | Attributes | SW 구성요소 이름 또는 식별자 |
| **버**전 문자열 (Version String) | Attributes | SW Version 정보 (Semantic Versioning) |
| **해**시 (Component Hash) | Attributes | SW 컴포넌트 해시 값을 통한 무결성 증빙 |
| **고**유 식별자 (Unique Identifier) | Attributes | 고유한 Namespace 및 고유 식별자 생성 |
| **관**계 (Relationship) | Attributes | SBOM 구성 요소 간의 종속성 및 연간 관계 표현 |

---

### 📊 SBOM 개념도

```
                    ┌──────────────┐
                    │ Bingo Buffer │
                    │    V2.2      │
                    └──────┬───────┘
                           │ Included in
                           ▼
                    ┌──────────────────┐
┌─────────────┐     │      Acme        │
│    Carol    │     │   Application   │
│ Compression │────▶│      V1.1       │
│    V3.1     │     └──────────────────┘
└─────────────┘              ▲
                             │ Included in
                    ┌────────┴───────┐
                    │  Bob Browser   │
                    │     V2.1       │
                    └────────────────┘
```

**구성요소 상세 정보:**

| Component Name | Supplier | Version | Author | Hash | UID | Relationship |
|:---------------|:---------|:--------|:-------|:-----|:----|:-------------|
| Application | Acme | 1.1 | Acme | 0x123 | 234 | Primary |
| ..Browser | Bob | 2.1 | Bob | 0x223 | 334 | Included in |
| ..Compression | Carol | 3.1 | Acme | 0x323 | 434 | Included in |
| ..Buffer | Bingo | 2.2 | Acme | 0x423 | 534 | Included in |

---

### 📋 표준 포맷 `S-C-S`

| 포맷 | 설명 | 특징 |
|:-----|:-----|:-----|
| **S**PDX | Software Package Data Exchange | 리눅스 재단 오픈소스 저작권 및 라이선스 정보 교환 표준 |
| **C**ycloneDX | - | OWASP 재단 공급망 구성요소 보안 및 경량 SBOM 표준 |
| **S**WID | Software Identification | SW 정보에 대한 Tag 생성 및 오픈소스 SW 인벤토리 지원 |

---

### 🔧 SBOM 고려사항 `관기`

| 구분 | 고려사항 | 내용 |
|:-----|:---------|:-----|
| **관**리적 | SBOM 관리요소 설정 | 미국 국가표준기술연구소 SBOM 최소 관리 요소 가이드 (NIST SBOM Minimum Elements Guide), 데이터 필드, 자동화 지원, 지침 절차 |
| **관**리적 | SW 공급망 위험 평가 정기 수행 | SW 공급망 취약점, 악성코드 유입 위험 정기적 평가, 잠재적 취약성 식별 및 공격 사전 대응 |
| **기**술적 | SBOM 자동화 구현 | SBOM 자동 생성 및 기계 가독성 고려한 자동화 기술, SPDX, CycloneDX, SWID 이용한 표준 양식 적용 |
| **기**술적 | SBOM 오픈 플랫폼 | IoTcube 등 SBOM 자료생성 및 보안취약점 분석 수행 |

### 📈 SBOM 활용 분야

| 분야 | 활용 내용 |
|:-----|:---------|
| 공급망 보안 | 서드파티 컴포넌트 취약점 파악 |
| 라이선스 컴플라이언스 | 오픈소스 라이선스 준수 확인 |
| 취약점 관리 | CVE 기반 보안 취약점 추적 |
| 자산 관리 | SW 구성요소 인벤토리 관리 |

---

### 🔄 SBOM 적용 프로세스

```
SW 개발
    ↓
컴포넌트 식별 및 메타데이터 수집
    ↓
SBOM 자동 생성 (SPDX/CycloneDX/SWID)
    ↓
SBOM 검증 및 업데이트
    ↓
배포 시 SBOM 포함
    ↓
취약점 스캔 및 모니터링
```

### ⭐ 차별점 키워드

### ✅ 학습 체크리스트

- [ ] SBOM의 정의를 설명할 수 있다
- [ ] SBOM 구성요소 8가지를 암기할 수 있다 (`작시공구버해고관`)
- [ ] 표준 포맷 3가지(SPDX, CycloneDX, SWID)를 구분할 수 있다 (`S-C-S`)
- [ ] SBOM 개념도를 이해하고 구성요소 간 관계를 설명할 수 있다
- [ ] 관리적 고려사항 2가지를 설명할 수 있다
- [ ] 기술적 고려사항 2가지를 설명할 수 있다
- [ ] SBOM 활용 분야 4가지를 설명할 수 있다
- [ ] 2021년 미국 행정명령과 SBOM 제출 의무화를 설명할 수 있다
- [ ] SBOM 자동화 구현 방법을 설명할 수 있다
