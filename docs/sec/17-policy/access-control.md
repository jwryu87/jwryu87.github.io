# 접근통제 정책과 메커니즘
{: .fs-8 }

17.2 보안정책
{: .label .label-purple }

---

## 핵심 키워드

- **접근통제 정책**: DAC, MAC, RBAC
- **접근통제 메커니즘**: ACL, CL, SL

---

## 접근통제 정책

### 정책 비교

| 구분 | DAC (임의적접근) | MAC (강제적접근) | RBAC (역할기반) |
|:-----|:-----------------|:-----------------|:----------------|
| **개념** | 주체, 그룹의 신분에 근거한 권한부여 | 주체가 갖는 권한 근거한 접근허여 | 역할을 배정 및 접근권한 부여 |
| **특징** | 하나의 사용자 정보단위로 구성(ID) | 정보의 비밀성 근거, 보안레벨이 주어짐 | 권한을 논리적, 독립적 할당 및 회수 |
| **통제주체** | 시스템 | 객체 오너, 관리자 | 중앙관리자 |
| **고려사항** | ID도용의 위험 | 성능및 구현문제 | 역할/권한 이력관계 |

### 정책 상세 비교

| 구분 | DAC (임의적접근) | MAC (강제적접근) | RBAC (역할기반) |
|:-----|:-----------------|:-----------------|:----------------|
| **부여방법** | Access Control/ Capability List | 자원의 중요도에 따른 보안등급(계층) | 계층적 역할분배, 최소권한 정책 |
| **방법예시** | [사람A]에 대하여 {프린터1:P}{파일1:RW} | '0'= 비보안, '1'=비밀,'2'=일급기밀 | 자장이 사람에게 파일읽을 권한부여 |
| **활용** | SSO 그룹웨어 인증 | 군사용, 보안업체 | ERP 패키지등 |

---

## 접근통제 메커니즘

| 메커니즘 | 설명 |
|:---------|:-----|
| **ACL (Access Control List)** | 주체가 객체에 어떤 행위를 할 수 있는지 표현하는 메커니즘 |
| **CL (Capability List)** | 주체에 대해 저장된 접근허가 목록을 관리하는 메커니즘 |
| **SL (Security Label)** | 객체에 부여된 보안 속성 정보의 집합을 통해 접근 |

---

## 접근통제 정책 상세

### DAC (Discretionary Access Control)

{: .note }
> 데이터 소유자가 접근 권한을 결정하는 가장 유연한 모델

| 특징 | 설명 |
|:-----|:-----|
| **권한 부여 주체** | 객체의 소유자 |
| **유연성** | 높음 |
| **관리 복잡도** | 중간 |
| **대표 구현** | Unix 파일 퍼미션, ACL |
| **단점** | 트로이 목마 공격에 취약 |

### MAC (Mandatory Access Control)

{: .note }
> 시스템이 보안 레벨에 따라 강제로 접근을 통제하는 모델

| 특징 | 설명 |
|:-----|:-----|
| **권한 부여 주체** | 시스템 (보안 관리자) |
| **유연성** | 낮음 |
| **관리 복잡도** | 높음 |
| **대표 구현** | Bell-LaPadula, Biba |
| **장점** | 정보 유출 방지에 효과적 |

### RBAC (Role-Based Access Control)

{: .note }
> 역할에 기반하여 권한을 부여하는 효율적인 모델

| 특징 | 설명 |
|:-----|:-----|
| **권한 부여 주체** | 역할 관리자 |
| **유연성** | 중간 |
| **관리 복잡도** | 낮음 |
| **대표 구현** | ERP, 기업 시스템 |
| **장점** | 최소 권한 원칙 구현 용이 |

---

## 정책 선택 기준

| 요구사항 | 적합한 정책 |
|:---------|:------------|
| 유연한 권한 관리 필요 | DAC |
| 높은 보안 수준 필요 | MAC |
| 대규모 조직 권한 관리 | RBAC |
| 규정 준수 필요 | MAC + RBAC |
| 최소 권한 원칙 적용 | RBAC |
