## 정의/개념

데이터 수신 측에서 오류 검출 후 송신측에 오류 사실을 알리고 재전송을 요청하여 오류를 수정하는 **네트워크 오류제어 기술** 및 **데이터베이스 보호 기술**

---

## DB 접근제어 시스템 종류

| 구분 | 설명 |
|:-----|:-----|
| **에이전트 방식** | - DB 직접 접근하여 모든 권한을 제어할 수 있고 감시함<br>- DB 서버 자체에 접근 제어 및 로깅 기능을 포함하는 에이전트 설치 |
| **게이트웨이(Proxy) 방식** | - 대상 DB 추가가 가능하여 대형 시스템 환경에 적절한 방식<br>- DB 어디에 접속하는 모든 IP를 프록시 서버를 경유하도록 설정<br>- 별도 |
| **인라인(In-Line) 방식** | - 규모가 크지 않고, DB 서버가 한 장소에 위치한 중소형에 적절한 방식<br>- 대상 DB 서버와 클라이언트 네트워크 사이에 In-Line 방식으로 구성함 |
| **스니핑 방식** | - TAP(Test Access Port)장비, 패킷 미러링을 통하여 DB 통신이 이루어지는 네트워크 패킷을 분석하여 사후 분석하는 감시 역할에 비용 줄임 |

---

## DB 암호화 방식 종류

### API 방식

```
┌────────────────┐     ┌────────────────┐
│ 어플리케이션 서버 │     │    DB 서버     │
│  ┌──────────┐  │     │  ┌──────────┐  │
│  │응용프로그램│  │     │  │   DBMS   │  │
│  └────┬─────┘  │     │  └──────────┘  │
│       ↓        │     │                │
│  ┌──────────┐  │     │                │
│  │암복호화 모듈│  │─────│                │
│  └──────────┘  │     │  암호화 데이터  │
└────────────────┘     └────────────────┘
```

- 암복호 모듈이 API 라이브러리 형태로 애플리케이션 서버에 설치되고, 응용프로그램에서 암복 모듈을 호출하는 방식
- 응용프로그램 수정이 불가피함

### Plug-in 방식

```
┌────────────────┐     ┌────────────────┐
│ 어플리케이션 서버 │     │    DB 서버     │
│  ┌──────────┐  │     │  ┌──────────┐  │
│  │응용프로그램│  │     │  │   DBMS   │  │
│  └──────────┘  │     │  │ EXTPROC  │  │
│                │     │  │  PL/SQL  │  │
│                │─────│  └────┬─────┘  │
│                │     │       ↓        │
│                │     │  ┌──────────┐  │
│                │     │  │  View    │  │
│                │     │  │  Trigger │  │
│                │     │  └──────────┘  │
│                │     │  암복호화 모듈  │
│                │     │  암호화 데이터  │
└────────────────┘     └────────────────┘
```

- 암복호 모듈이 DB 서버에 설치, DBMS에서 Plug-In된 암복호화 모듈을 호출하는 방식
- 응용 프로그램 수정은 필요 없으나, 기존 DB 스키마에 대응하는 View 생성 및 암호화할 테이블이 필요하여 가 장애이 패요함

### TDE 방식 (Transparent Data Encryption)

```
┌────────────────┐     ┌────────────────┐
│ 어플리케이션 서버 │     │    DB 서버     │
│  ┌──────────┐  │     │  ┌──────────┐  │
│  │응용프로그램│  │     │  │   DBMS   │──│──암호프로세스
│  └──────────┘  │─────│  └──────────┘  │
│                │     │                │
│                │     │  암호화 데이터  │
└────────────────┘     └────────────────┘
```

- DBMS 커널 수준에서 처리되어 기존 응용 프로그램 수정 DB 스키마 변경의 필요 없음
- DBMS에 내장되어 있는 암호화 기능(TDE: Transparent Data Encryption)을 이용하는 방법

### OS 암호화 방식

```
┌────────────────┐     ┌────────────────┐
│ 어플리케이션 서버 │     │    DB 서버     │
│  ┌──────────┐  │     │  ┌──────────┐  │
│  │응용프로그램│  │     │  │   DBMS   │  │
│  └──────────┘  │─────│  └──────────┘  │
│                │     │       │        │
│                │     │  ┌────┴───┐    │
│                │     │  │  암호   │    │
│                │     │  │  API   │    │
│                │     │  └────┬───┘    │
│                │     │  암복호화 데이터│
└────────────────┘     └────────────────┘
```

- OS에서 I/O 발생 시 시스템 호출 시 DB 파일 자체를 암호화하는 방식
- 응용프로그램 수정, DB 스키마 수정이 필요 없지만, DB 파일 전체를 암호화하기 때문에 DB 서버 및 DB 서버에 큰 부하가 발생할 수 있음

---

## 암호화 방식 비교

| 방식 | 장점 | 단점 |
|:-----|:-----|:-----|
| **API** | 유연한 암호화 범위 지정 | 응용프로그램 수정 필요 |
| **Plug-in** | 응용프로그램 수정 없음 | View/Trigger 생성 필요 |
| **TDE** | 투명한 암호화 | DBMS 의존적 |
| **OS** | 완전한 투명성 | 서버 부하 발생 |
